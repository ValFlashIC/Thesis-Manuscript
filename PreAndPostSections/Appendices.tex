\paragraph{COO} The simplest \emph{sparse format} is called \emph{COO} (\emph{COOrdinate})~\cite{Tewarson1973_COO_CSR_LIL_first, Saad2003_COO_CSR_DIA_ELL, Chou2018_COO_CSR_DIA_ELL_HASH}.
It consists in three separate tables, one filled only with the non-zero elements of the matrix, a row index table containing the row of the corresponding elements, and similarly a column index table (see Figure~\ref{}).
The two latter are \emph{metadata} tables and all three tables are of size \emph{nnz}.
In this format, the elements are not necessarily sorted, the memory footprint does not depend on the structure of the matrix and all elements are stored independently of each other.

\paragraph{CSR \& CSC} The most commonly used \emph{sparse format} is called \emph{CSR} (\emph{Compressed Sparse Row})~\cite{Tewarson1973_COO_CSR_LIL_first, Barrett1994_CSR_BCSR_DIA_ELL, Saad2003_COO_CSR_DIA_ELL, Chou2018_COO_CSR_DIA_ELL_HASH}.
It is similar to the \emph{COO} format except that the row index table is replaced by a set of indices that point to the position of the start of each row in the column index table (see Figure~\ref{}).
The size of this table is the number of rows plus one, and, in practical cases, the memory footprint is smaller than with \emph{COO}.
However, the non-zero elements need to be sorted in \emph{row-wise} order.
This format has a \emph{column-wise} variant called \emph{CSC} (\emph{Compressed Sparse Column}) where the column index table stores the positions of columns instead of rows.
These two formats are symmetrical but with \emph{CSR}, it is the traversal of rows that is efficient because they are sequential whereas in \emph{CSC}, column traversal is efficient.

\paragraph{BCSR} Even in sparse matrices, there can be regions, or blocks, that have a large fraction of non-zero values.
In this case, the \emph{BCSR} format~\cite{Barrett1994_CSR_BCSR_DIA_ELL}, which is an extension of the \emph{CSR} format with \emph{tiling}, can be attractive.
We use the term \emph{tiling} to describe a level of hierarchy in a storage format.
That is, a \emph{tile} refers to a sub-matrix, which itself may be dense, as in the case of \emph{BCSR}, or which could be sparse.
With the \emph{BCSR} format, we simply replace the non-zero elements of the \emph{CSR} format with dense, fixed-size sub-matrices (see Figure~\ref{}).
Within these sub-matrices, it is possible to have some zero elements.
The advantage of the \emph{BCSR} format is that the dense sub-matrices can be processed using techniques optimized for dense matrices, particularly using vector units or GPUs~\cite{Im2004_TilingBCSR}.
Compared to the \emph{CSR} format, \emph{BCSR} makes it possible to accelerate the computation within the dense \emph{tiles}, but this comes with the overhead of storing some zero values.
Not all matrices are well suited to the \emph{BCSR} format.

\paragraph{DIA} For diagonally structured matrices
($\forall(i, j) \in \ldb0, M\rdb \times \ldb0, N\rdb, \; a_{i,j}=0 \; for \; \lvert i-j \rvert > K$), a format called \emph{DIA} (\emph{DIAgonal}) is particularly well suited~\cite{Barrett1994_CSR_BCSR_DIA_ELL, Saad2003_COO_CSR_DIA_ELL, Chou2018_COO_CSR_DIA_ELL_HASH}.
It consists in storing the values along the $2 \times K+1$ diagonals in a table and keeping information about the position of the diagonals in the matrix (see Figure~\ref{}).
The organization of the non-zero data and selection of the diagonals can be tailored to the matrices and access patterns.
This format is well suited to diagonal matrices, has a low overhead for storing indices and can be combined with other formats to better fit more complex structures.

\paragraph{ELL} Another format called \emph{ELL} (\emph{ELLPACK})~\cite{Kincaid1989_ELLPACK, Barrett1994_CSR_BCSR_DIA_ELL, Saad2003_COO_CSR_DIA_ELL, Chou2018_COO_CSR_DIA_ELL_HASH} is well suited when the number of non-zero elements per row can be bounded by $K$.
In this case, the non-zero values in a $M \times N$ matrix can be stored as a dense $M \times K$ matrix, augmented with an index table, of the same dimensions, which gives the column position of each non-zero value (see Figure~\ref{}).
This format is not efficient if the number of non-zero elements per row is highly variable, as many entries in the $M \times K$ table are unused.
This format is well suited to GPUs~\cite{Vazquez2009_ELLforGPUs, Cao2010_ELLforGPUs} since it transforms a sparse matrix into a dense one in memory.

\paragraph{HYB} The \emph{HYB} format (HYBrid)~\cite{Bell2009_HYB} is an example of a format which combines \emph{ELL} and
\emph{COO}.
The \emph{ELL} format can be used, but with a value of $K$ (the width of the value table) that is smaller than the maximum non-zero entries per row in the 
original matrix (see Figure~\ref{}).
For the small number of rows where the table lacks a sufficient number of columns, the \emph{COO} format can be used to store the extra values.
In this way, the storage efficiency of \emph{ELL} is improved, and it can be applied to a broader class of matrices.

\label{app:sparse_formats_rep}

A visual representation of compressed matrices, called \emph{FiberTree}~\cite{Sze2020_FiberTree}, is helpful to understand these \glspl{spFormat}. Each \emph{rank} of a matrix represents a level of a tree where the nodes are filled with either the \gls{metadata} of the child nodes or the matrix data which can only appear in the leaves (see Figure~\ref{fig:app:FiberTreeExCSR}). With this representation, we clearly identify the number of \emph{ranks}, which define the \emph{fibers}. A \emph{fiber} is the generic term to describe a one-dimensional stream of data or \gls{metadata}, which corresponds to a row or column in the case of a two-dimensional matrix. A stream of indices is a \emph{fiber} of \gls{metadata}.% in glossary !!
Figure~\ref{fig:app:FiberTreeExCSR} presents, as an example, a \emph{FiberTree} representation of the matrix of Figure~\ref{fig:bg:FormatEx:Dense}, matching the \acrshort{csr} format: each \emph{rank} in the tree corresponds to a \emph{fiber} stored in the table \emph{Row PTR} or \emph{Col IDX} of Figure~\ref{fig:bg:FormatEx:CSR}.

In a recent work~\cite{Wu2022_SparseLoop}, a systematic nomenclature for \emph{sparse matrix (and tensor) formats} was proposed, based on \emph{FiberTree} representation. In fact, in the example of Figure~\ref{fig:app:FiberTreeExCSR}, the \emph{fibers} are compressed with different methods. Thus each \emph{rank} can be characterized by a label indicating how it is compressed: \emph{Uncompressed} (\emph{U}), \emph{Coordinate Payload} (\emph{CP}), \emph{Uncompressed Offset Pairs} (\emph{UOP}).
\emph{U} defines \emph{ranks} where all data values are stored contiguously in memory. \emph{CP} defines \emph{ranks} that store the indices of each payload, a payload being either a non-zero element or a sub-matrix\footnote{In this section, we use the term sub-matrix but this is extended to the notion of \gls{tiling} in Section~\ref{sec:bg:tiling}.}. \emph{UOP} defines \emph{ranks} that store pointers (offsets) to delimit sub-matrices of the next \emph{rank}\footnote{If some sub-matrices are empty, pointers will be repeated, hence 'uncompressed'.}.
These are the three main ones as they are used in the most common \glspl{spFormat} but many others exist or can be created.

With this nomenclature, the \glspl{spFormat} can be classified according to their \emph{rank} compression methods. For example, the \acrshort{csr} format is denoted as \emph{(UOP, CP) row-major} since its \emph{Row PTR} table stores pointers (offsets) to rows. The \acrshort{csr} and \acrshort{csc} formats are both of type \emph{(UOP, CP)}, one being \emph{row-major}, the other \emph{column-major}. The \acrshort{coo} format is denoted as $CP^2$ because there are two coordinates (row and column) for each non-zero element, as it is a \emph{CP fiber} having two indices per data element instead of one.

User defined formats can also be easily described and named. For example, a user could define a storage format with a nested \acrshort{csr}. That is, each non-zero element in the outer \acrshort{csr}, corresponds to a sub-matrix, itself stored in \acrshort{csr} format. Such a representation would be labeled \emph{(UOP, CP, UOP, CP)}.
A similar nomenclature is proposed by TACO~\cite{Chou2018_COO_CSR_DIA_ELL_HASH}. It is more adapted to code generation applications, where the \emph{FiberTree}-based nomenclature better describes the memory storage of the \gls{spFormat}. For example, \emph{U} corresponds to \emph{Dense}, \emph{CP} to \emph{Singleton}, the pair \emph{(UOP, CP)} is translated as \emph{Compressed} while \emph{UOP} alone corresponds to \emph{Offset}. We chose to adopt the \emph{FiberTree}-based nomenclature in this article since it is more suited to a hardware context.
