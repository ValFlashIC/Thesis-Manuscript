\section{High Level Evaluation of SpDCache}
\label{sec:spdc:eval}

\lettrine{W}{e} modelled the behavior of SpDCache, based on the flowchart in Figure~\ref{fig:spdc:spdc_archi}, using a transaction level model in Python, for a single-core, single-thread system executing a \acrshort{op} \acrshort{spmv} kernel (Algorithm~\ref{alg:spdc:spmv_with_red}).
We do not model timing, as the order of the memory accesses determines the behavior of the cache (\glspl{hit}, \glspl{miss}, \glspl{eviction}), which depends only on the \acrshort{spmv} algorithm.
The model checks the final numerical result and reports the main memory traffic associated with the \emph{updates} to the output vector (\emph{Output Memory Traffic}), where we count the bytes transferred (64B for each read and write, and 8B per atomic \acrshort{rmw}, with our configuration).
For each write transaction, it tracks how many words in the line were actually used (\emph{Evicted Cache Line Utilization}) and it tracks how many words in the cache held useful data (\emph{Cache Memory Utilization}).
We consider a word in the cache to be useful if it is updated or inserted by the processor.
Our goal is to validate the principle of SpDCache using real application matrices from the SuiteSparse Matrix Collection~\cite{Kolodziej2019_SuiteSparse}.

As baselines, we implemented a fully generic cache (\acrshort{gc}) that handle reads and writes for both inputs ($A$, $b$) and output ($c$), following the data movements shown in Figure~\ref{fig:spdc:GC_cache_WB}, and a simple \gls{redC} (\acrshort{redc}) that processes \emph{RED} transactions in a dedicated region using the flow shown in Figure~\ref{fig:spdc:red_based_cache}.
In the \acrshort{redc}, 25\% of the memory is allocated for the inputs and the other 75\% for \emph{REDs} on $c$.
In SpDCache (\acrshort{spdc}), 25\% is allocated for the \acrshort{grc}, 50\% for the \acrshort{drc} and 25\% for the \acrshort{srt}.
All three caches use a \emph{random} eviction policy and have a total data storage of 32~kiB, and we do not account for the \emph{TAG} storage.
The \gls{set}-associative regions have 4~\glspl{way} and 64B cache-lines (8 \emph{double} words per line).

We simulated 48 matrices that have been used by the state-of-the-art accelerators~\cite{Pal2018_OuterSPACE, Hegde2019_ExTensor, Srivastava2020_MatRaptor, Baek2021_InnerSP, Zhang2021_Gamma, Srikanth2021_SortCache, Zhang2022_ASA}.
In Table~\ref{tab:spdc:results}, we report the results for a representative subset; the gains with the full \gls{set} are similar.
Most, but not all, of the matrices have a dense \gls{band} and even some matrices without an obvious \gls{band} benefit from the blocking effect in the \acrshort{drc}.

As expected, the simple \gls{redC} (\acrshort{redc}) decreases the main memory traffic to 39.2\% of the \acrshort{gc} baseline.
SpDCache has significantly better performance than the \acrshort{redc}, reducing the traffic to 12.3\%.

The benefits of SpDCache can be partly explained by the improved utilization of the cache-lines.
On average, a line evicted from the \acrshort{drc} has 7.46 modified words, compared to 3.69 and 3.91 with the baselines.
This improved utilization is the result of blocking the dense region in the \acrshort{drc}.

We need to consider an additional metric, the percentage of traffic sent as atomic \acrshort{rmw} transactions (\textit{\%~ATO}).
If this value is too high, it means we have simply shifted the workload to the lower level caches, which inherently have limited compute capability.
This is the case for the \emph{msc10848} matrix where 70.9\% of the traffic is atomic \acrshortpl{rmw}, where a very large fraction of the non-zero values are outside the \gls{band}.
Conversely, the \emph{raefsky3} matrix is an example were no atomic \acrshortpl{rmw} are used, as the region outside the \gls{band} was empty.
In this case, 25\% of the available memory was unused.

To facilitate comparison with other accelerators, we simulated the \gls{set} of matrices used by OuterSPACE~\cite{Pal2018_OuterSPACE}, a commonly used baseline (not shown in Table~\ref{tab:spdc:results}).
On these matrices, SpDCache's memory traffic is 8.8$\times$ lower than \acrshort{gc}.

\begin{highlight}{Conclusion}{Red}
    SpDCache achieves a 8.1$\times$ \textbf{decrease in memory traffic} on the output vector $c$ compared to a generic cache, over a \gls{set} of 48 matrices.
    It also \textbf{outperforms a simple \gls{redC}} by 3.8$\times$, showing the importance of having separate dedicated cache regions for the in- and out-of-band of a matrix.
\end{highlight}
