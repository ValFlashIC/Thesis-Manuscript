\section{High Level Evaluation of SpDCache}
\label{sec:spdc:eval}

\lettrine{W}{e} modelled the behavior of \acrlong{spdc}, based on the flowchart in Figure~\ref{fig:spdc:spdc_archi}, using a transaction level model in Python, for a single-core, single-thread system executing a \acrshort{op} \acrshort{spmv} kernel (Algorithm~\ref{alg:spdc:spmv_with_red}).
We do not model timing, as the order of the memory accesses determines the behavior of the cache (\glspl{hit}, \glspl{miss}, \glspl{eviction}), which depends only on the \acrshort{spmv} algorithm.
The model checks the final numerical result and reports the main memory traffic associated with the \emph{updates} to the output vector (\emph{Output Memory Traffic}), where we count the bytes transferred (64B for each read and write, and 8B per atomic \acrshort{rmw}, with our configuration).
For each write transaction, it tracks how many words in the line were actually used (\emph{Evicted Cache Line Utilization}) and it tracks how many words in the cache held useful data (\emph{Cache Memory Utilization}).
We consider a word in the cache to be useful if it is updated or inserted by the processor.
Our goal is to validate the principle of \acrlong{spdc} using real application matrices from the SuiteSparse Matrix Collection~\cite{Kolodziej2019_SuiteSparse}.

As baselines, we implemented a fully generic cache (\acrshort{gc}) that handle reads and writes for both inputs ($A$, $b$) and output ($c$), following the data movements shown in Figure~\ref{fig:spdc:GC_cache_WB}, and a simple \gls{redC} (\acrshort{redc}) that processes \emph{RED} transactions in a dedicated region using the flow shown in Figure~\ref{fig:spdc:red_based_cache}.
In the \acrshort{redc}, 25\% of the memory is allocated for the inputs and the other 75\% for \emph{REDs} on $c$.
In \acrlong{spdc} (\acrshort{spdc}), 25\% is allocated for the \acrshort{grc}, 50\% for the \acrshort{drc} and 25\% for the \acrshort{srt}.
In the next section, we present more results with varying region sizes, showing that this configuration is a good compromise.
All three caches use a \acrfull{lru} eviction policy, except the \acrshort{srt} region of \acrshort{spdc} which uses a \emph{random} policy.
All three caches have a total data storage of 32~KiB, and we do not account for the \emph{TAG} storage.
The \gls{set}-associative regions have 4~\glspl{way} and 64B cache-lines (8 \emph{double} words per line).

We simulated 48 matrices that have been used by the state-of-the-art accelerators~\cite{Pal2018_OuterSPACE, Hegde2019_ExTensor, Srivastava2020_MatRaptor, Baek2021_InnerSP, Zhang2021_Gamma, Srikanth2021_SortCache, Zhang2022_ASA}, which are listed in Appendix~\ref{app:mat_list}.
We excluded matrices whose output vector $c$ fits entirely in the cache (fewer than 4096 elements), since they do not stress the cache.
In Table~\ref{tab:spdc:results}, we report the results for all three caches, along with statistical indicators.
Most, but not all, of the matrices have a dense \gls{band} and even some matrices without an obvious \gls{band} benefit from the blocking effect in the \acrshort{drc}.

As expected, the simple \gls{redC} (\acrshort{redc}) decreases the main memory traffic to 89.2\% of the \acrshort{gc} baseline.
\acrlong{spdc} has significantly better performance than \acrshort{redc}, reducing the traffic to 24.5\%.
\acrlong{spdc} thus achieves a 4.1$\times$ decrease in memory traffic compared to \acrshort{gc}, and a 3.6$\times$ decrease compared to \acrshort{redc}, on average over the set of matrices.
\acrshort{spdc} outperforms \acrshort{gc} on all matrices, and \acrshort{redc} on 93.8\% of them.

The benefits of \acrlong{spdc} can be partly explained by the improved utilization of the cache-lines.
On average, a cache-line evicted from \acrshort{spdc} has 7.21 modified words, compared to 3.56 and 3.60 with the baselines.
This improved utilization is the result of blocking the dense region in the \acrshort{drc}, where multiple \glspl{reduction} have the time to accumulate in the cache before being evicted.
The \acrshort{srt} also contributes to this effect, as it captures irregular accesses outside the \gls{band}, which would otherwise evict useful data from the \acrshort{drc}.
The overall cache memory utilization is also significantly improved, with 75.2\% of the cache holding useful data in \acrshort{spdc}, compared to 45.8\% and 36.4\% for \acrshort{redc} and \acrshort{gc}, respectively.

We need to consider an additional metric, the percentage of traffic sent as atomic \acrshort{rmw} transactions (\textit{\%~\acrshort{rmw}}).
If this value is too high, it means we have simply shifted the workload to the lower level caches, which inherently have limited compute capability.
In mean, 40.5\% of the memory traffic of \acrlong{spdc} is composed of atomic \acrshortpl{rmw}, which is acceptable given the significant overall reduction in memory traffic.
However, this ratio varies widely between matrices.
There is some matrices where most of the traffic is composed of atomic \acrshortpl{rmw}, meaning that most of their non-zero values are outside the \gls{band}.
For such matrices, which are basically not banded, \acrshort{spdc} still greatly decrease the memory traffic, but better performance could be achieved by choosing a matrix-fitting region partitioning strategy.
For example, the matrix \emph{cit-Patents} has no elements in the \gls{band}, and all non-zero values are scattered under the bottom part of the matrix (an unusual extreme case).
While having a memory traffic decreased to 7.2\% compared to \acrshort{gc}, it could benefit from a better use of the \acrshort{drc} by choosing a horizontal band covering the non-zero elements, with correct sizing to leverage the blocking effect.
On the other hand, some matrices have very low \acrshort{rmw} usage, indicating that most non-zero values are within the \gls{band} and efficiently handled by the \acrshort{drc}.
In this case, 25\% of the available memory was unused.
The \acrshort{srt} size could be adjusted to gain performance.

\begin{highlight}{Conclusion}{Red}
    \acrlong{spdc} achieves a 4.1$\times$ \textbf{decrease in memory traffic} on the output vector $c$ compared to a generic cache, over a set of 48 matrices.
    It also \textbf{outperforms a simple \gls{redC}} by 3.6$\times$, showing the importance of having separate dedicated cache regions for the in- and out-of-band of a matrix.
\end{highlight}

\section{State-of-the-Art Comparison and Region Sizing}
\label{sec:spdc:soa_comp}

\lettrine{T}{o} compare \acrlong{spdc} with the state-of-the-art, we simulated PHI~\cite{Mukkara2019_PHI}, the existing solution that is closest to our proposal, using the same methodology as in the previous section.
PHI is a \gls{reduction}-aware cache that uses a single region to process \emph{RED} transactions (see Section~\ref{sec:soa:phi} from previous chapter).
PHI works as if our three regions, \acrshort{grc}, \acrshort{drc} and \acrshort{srt}, where merged together, using different technics to handle \glspl{reduction}, load and store operations in the same cache address space.
It also handles isolated elements, but do not consider the matrix structure.
Conversely to \acrshort{spdc}, PHI relies on the use of the \emph{deferred update} version of the \acrshort{op} \acrshort{spmv} algorithm (see Algorithm~\ref{alg:bg:OPSpMSpMCSR}).
We thus adapted our simulation to use this algorithm for PHI.
To ensure a fair comparison, we allocated the same amount of memory to PHI as for \acrshort{spdc} and the baselines (32~KiB).
In Table~\ref{tab:spdc:results_overall}, we report the results for PHI alongside our previous results (columns 2, 3, 6 and 7).

As expected, PHI achieves a significant reduction in memory traffic compared to the \acrshort{gc}, down to 50.4\% on average.
This represents a 2.0$\times$ decrease compared to the \acrshort{gc}, which is consistent with the results reported in the original PHI paper~\cite{Mukkara2019_PHI}, and a 1.8$\times$ decrease compared to the \acrshort{redc}.
However, \acrlong{spdc} still outperforms PHI, cutting memory traffic by 2.1$\times$.
The improved performance of \acrshort{spdc} can again be explained by the better utilization of the cache-lines, with 7.21 useful words per evicted line, compared to 5.00 for PHI.
The overall cache memory utilization is also significantly better with \acrshort{spdc}, with 75.2\% of the cache holding useful data, compared to only 30.6\% for PHI.

We also explored different region sizing for \acrlong{spdc}, varying the sizes of the \acrshort{grc}, \acrshort{drc} and \acrshort{srt} while keeping the total cache size constant at 32~KiB.
The results are also reported in Table~\ref{tab:spdc:results_overall}.
Reducing the size of the \acrshort{grc} from 8~KiB (32 \glspl{set}) to 2~KiB (8 \glspl{set}) and 1~KiB (4 \glspl{set}) has little impact on performance, as the \acrshort{grc} is mainly used for temporary storage of input vector elements.
However, reducing it under 1~KiB would increase the total memory traffic, as more frequent evictions would occur before the data is used, an effect already noticeable when going from 2~KiB to 1~KiB.
With constant \acrshort{grc} size, when reducing the size of the \acrshort{srt}, which means allocating more space to the \acrshort{drc}, we observe a slight increase in memory traffic, as the \acrshort{srt} becomes less effective at capturing irregular accesses.
Thus, the \acrshort{srt} sizing appears to be more critical than the \acrshort{drc} sizing.
We will explore the reasons behind this effect in the next chapter.
Overall, fine tuning the region sizes can lead to small performance improvements, but the original sizing of 8~KiB for the \acrshort{grc}, 16~KiB for the \acrshort{drc} and 8~KiB for the \acrshort{srt} appears to be a good compromise for the set of matrices we evaluated.

About the \acrshort{redc}, we also explored different sizing strategies.
We observed that allocating more than 75\% of the cache (96 \glspl{set}) to store \emph{RED} transactions slightly improves performance, as it increases the likelihood of capturing multiple updates to the same output element before eviction, without overly compromising the storage of input elements.

\begin{highlight}{Conclusion}{Red}
    \textbf{\acrlong{spdc} outperforms PHI}, a state-of-the-art \gls{reduction} cache, by achieving a 2.1$\times$ decrease in memory traffic, thanks to its dedicated regions for in- and out-of-band accesses that improve cache-line and overall cache utilization.
    Additionally, we found that the simple 25/50/25\% region sizing of \acrlong{spdc} provides good performance with only minor variations observed when adjusting the sizes of the \acrshort{grc}, \acrshort{drc}, and \acrshort{srt}.
\end{highlight}
