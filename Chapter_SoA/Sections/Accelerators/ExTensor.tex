\subsection{ExTensor}
\label{sec:soa:extensor}

The authors created ExTensor~\cite{Hegde2019_ExTensor} as a general purpose sparse tensor algebra accelerator (including \acrshort{spmspm}).
It is an \acrshort{ip} accelerator but due to its support for three levels of fixed \gls{tiling} of the inputs, it optimizes the \gls{intersection} search between non-zero elements or \glspl{tile}.
By doing the \glspl{intersection} ahead of time, at different levels of the memory hierarchy (which correspond to the \gls{tile} levels), ExTensor is able to efficiently eliminate \glspl{innefOp}, thus addressing the main issue in \acrshort{ip} or complex \gls{tiling} algorithms.
Note that the input matrices would typically be stored in a complex sparse format derived from \acrshort{csr}, corresponding to three levels of \gls{tiling}.
Thus, at a given \gls{tile} level, each \gls{tile} is represented by coordinates (\textit{coord}), the \acrshort{csr} \gls{metadata} stored in $idx$ and $ptr$, with the \gls{tile} data itself compressed using \acrshort{csr} (next \gls{tile} level).
ExTensor is designed with a flexible architecture composed of different bricks, as shown in Figure~\ref{fig:soa:extensor}.
To aid in the understanding, the interactions between the bricks are described using the semantics of function calls.
\begin{itemize}
    \item The \textit{Scanner} is a storage unit that delivers a stream of coordinates in increasing order, fetching them from memory.
    The coordinates are delivered through the \textit{Iterate()} operation call.
    %
    \item The \textit{Intersect} is a hardware block that co-iterates over several \textit{Scanners} in parallel and compares the coordinate streams.
    This unit performs the \gls{intersection} search required by \acrshort{ip} type algorithms, delivering a single stream with the matching coordinates, which are the coordinates of non-zero \glspl{psum}.
    To optimise the \gls{intersection} step, it is possible to skip a range of coordinates in a stream depending on the current coordinate of another stream.
    Thus, the \textit{Intersect} sends skip messages with \textit{SkipTo()} operations to the other \textit{Scanners}.
    %
    \item The \textit{Coordinator} is a hardware unit that includes \textit{Scanners} and \textit{Intersect} units, and manages the input and output data depending on the current processed \gls{tile}, as shown in Figure~\ref{fig:soa:extensor}.
    Data and \gls{metadata} are respectively stored in storage units and \textit{Scanners}, while a \textit{Sequencer} manages the \textit{Iterate()} calls so that the \textit{Intersect} brick delivers the stream of effective coordinates.
    In addition, in the \textit{Tile Coord} brick, the offsets of the \gls{tile} are added back, so the output of the \textit{Coordinator} is in absolute coordinates.
\end{itemize}

These bricks can be placed at different positions in a memory hierarchy, depending on the \gls{tiling} and selected hardware.
Thus, the \textit{Coordinator} is able to intersect at coarse-granularity with output data being \gls{metadata} of the next \gls{tile}, skipping an entire \gls{tile} when appropriate, with low computation cost, and also at fine-granularity when output data are the actual non-zero elements that need to be multiplied.

ExTensor is designed with three levels of \gls{tiling} such that each have either input or output sequentiality.
First, a \textit{Sequencer} plus \textit{Scanners} block~\ding{192} is placed close to the main memory to determine which \glspl{tile} to send to the next storage, the \textit{Last Level Buffer} (\textit{LLB}).
Then, for each \gls{tile}, a \textit{Coordinator}~\ding{193} in the \textit{LLB} intersects the next level \glspl{tile}, which are sent to several \acrshortpl{pe} that will process in parallel.
Finally, in each \acrshort{pe}, a \textit{Coordinator}~\ding{194} intersects the non-zero elements of the computed \gls{tile} and sends them to the arithmetic unit of the \acrshort{pe} for multiplication.

In addition to these bricks which are the main contributions, ExTensor still needs to perform the \textit{merge} phase, including the \gls{reduction} of the \acrshortpl{po} which are stored in \acrshort{coo} format.
The \textit{Partial Output Buffer} (\textit{POB}), is managed by \glspl{tile}, and the content of the \gls{tile} is stored on-chip using a \acrfull{cam} for quick access, enabling \glspl{reduction} to be performed.
On overflow, the \glspl{tile} are flushed to main memory.
To benefit from ExTensor's support for \gls{tiling}, the input matrices need preprocessing, making it difficult to directly compare performance with other accelerators.

\begin{highlight}{Conclusion}{Red}
    \textbf{ExTensor} is a versatile, \acrshort{ip}-based sparse tensor algebra accelerator that optimizes \gls{intersection} searches through a three-level \gls{tiling} hierarchy, enabling efficient elimination of \glspl{innefOp} and supporting both coarse and fine-grain parallelism.
\end{highlight}
