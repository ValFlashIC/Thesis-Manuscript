\section{Hardware Challenges for Sparse Matrix Computation}
\label{sec:bg:hwchallenges}

\lettrine{I}{n} the previous section we presented a brief background on matrix multiplication without considering the underlying hardware.
Although the various \glspl{spFormat} reduce the memory footprint, they have the disadvantage that indirect accesses are required, which creates data-dependencies and reduces the effectiveness of caches.
As we have seen, with all three algorithms, at least some of the accesses are irregular.
The lack of spatial locality for these accesses reduces the benefit of having caches~\cite{Wulf1995_MemoryWall, Matam2012}.
In other words, a full cache-line is loaded, but only a few entries are read or modified.
Moreover, the matrices of interest are extremely large and even the non-zero elements, and associated \gls{metadata}, can not generally fit in the cache.
Depending on the algorithm, successive accesses to the same element may be too far apart to benefit from temporal locality in the cache.

\subsection{Inner-Product Algorithm}

In an \acrshort{ip} algorithm, the $A$-\gls{metadata} needs to be read to indirectly access the $B$-matrix (or $b$-vector), which may itself be stored in a \gls{spFormat}.
In this case, the algorithm needs to check if the non-zero $A$-indices are present in $B$ (or $b$).
If an index appears in neither list, there is no need to perform the multiplication (\gls{innefOp}).
More generally, this problem applies to any \gls{fiber} and is called the \gls{intersection} search.
As a minimum, this requires reading all the \gls{metadata} for $A$ and $B$ (or $b$), and identifying those present in both.

In Algorithm~\ref{alg:bg:IPSpMSpMCSR}, we present the pseudo-code for an \acrfull{ip} \acrshort{spmspm} with $A$ and $B$ stored in \acrshort{csr} and \acrshort{csc} formats, respectively.
The outer-loop ($M$, line 1) iterates over the rows of $A$.
The middle-loop ($N$, line 2) iterates over the columns of $B$.
The inner-loop (line 4) iterates over the non-zero elements of the current row ($i$) of $A$, and for each index ($k_A$ of position $pos_A$ in the $idx$ array of $A$, line 5), a search is performed in the \gls{metadata} of $B$ (line 6), to see if the corresponding index in $B$ is present.
Assuming it is ($\mathit{isMatch}$ is true, line 7), the \gls{psum} is computed and then accumulated locally ($acc$, line 8).
This example highlights the need to efficiently search for the \gls{intersection} of indices in the \gls{metadata} of the inputs.

Different algorithms can be used to address the \gls{intersection} search, depending on whether the elements are sorted.
When the elements are sorted, the \gls{intersection} becomes a \emph{merge} and generally requires at most $2 \times nnz - 1$ comparisons (in some cases the number of comparisons can be reduced~\cite{Baeza-Yates2010_Intersection}).
In Algorithm~\ref{alg:bg:IPSpMSpMCSR}, we use sorted \glspl{spFormat} (\acrshort{csr} and \acrshort{csc}).
Thus, the na\"ive \gls{intersection} search proposed (line 10) does at most one traversal of each the current row of $A$ ($i$) and the current column of $B$ ($j$).
Note that computing an \gls{intersection} in software induces the use of conditional loops and branches, which can be costly for classical CPU branch predictors.
In case the matrix $B$ (or vector $b$) is stored in a dense format, the \gls{intersection} search is simplified, since all indices are implicitly present, but the accesses to $B$ (or $b$) remain indirect.
As we see in Chapter~\ref{cha:soa}, certain accelerators provide support for performing the \gls{intersection} operation directly in hardware.

\begin{algorithm}[htbp]
    \caption{\acrshort{ip} \acrshort{spmspm} Performing \Gls{intersection} Searches with \acrshort{csr} and \acrshort{csc} Formats}
    \label{alg:bg:IPSpMSpMCSR}
    %
    \KwData{$A(M, K)$ in \acrshort{csr}: $\mathit{A.val}, \mathit{A.idx}, \mathit{A.ptr}$}
    \KwData{$B(K, N)$ in \acrshort{csc}: $\mathit{B.val}, \mathit{B.idx}, \mathit{B.ptr}$}
    \KwData{$C(M, N)$ in \gls{dFormat} (for readability)}
    \KwResult{$C = A \times B$}
    \BlankLine
    \For(\tcp*[f]{Iterations over the rows of A}){$i \in \ldb 0, M-1 \rdb$}{
        \For(\tcp*[f]{Iterations over the columns of B}){$j \in \ldb 0, N-1 \rdb$}{
            $acc \gets 0$; $pos_B \gets \mathit{B.ptr}_j$\;
            \For{$pos_A \in \ldb \mathit{A.ptr}_i, \mathit{A.ptr}_{i+1}-1 \rdb$}{
                \tcp{Iterations over the non-zero elements of the i-th row of A}
                $k_A \gets \mathit{A.idx}_{pos_A}$\;
                \tcp{Column-index of the (i, pos\textsubscript{A}) non-zero element of A}
                $(\mathit{isMatch}, pos_B) \gets \mathit{IntersectionSearch}(k_A, j, pos_B, \mathit{B.idx}, \mathit{B.ptr})$\;
                \tcp{Return true and the matching pos\textsubscript{B} if idx exists in the j-th column of B}
                \If(\tcp*[f]{If indices did match}){$\mathit{isMatch}$}{
                    $acc \gets acc + \mathit{A.val}_{pos_A} \times \mathit{B.val}_{pos_B}$\;
                    \tcp{Local partial sum accumulation}
                }
            }
            $C_{i,j} \gets acc$\tcp*[r]{Write in memory of the resulting (i, j) element of C}
        }
    }
    \BlankLine
    \Fn{$\mathit{IntersectionSearch}(k_A, j, pos_B, \mathit{B.idx}, \mathit{B.ptr})$}{
        \tcp{Naive intersection search allowing the i-th row of A and the j-th column of B to be traversed only once at iteration (i, j)}
        \While{$\mathit{B.idx}_{pos_B} < k_A$ \textbf{and} $pos_B < \mathit{B.ptr}_{j+1}$}{
            $pos_B \gets pos_B + 1$\;
        }
        \If(\tcp*[f]{Returns true if indices from A and B match}){$\mathit{B.idx}_{pos_B} = k_A$}{
            \textbf{return} $(true, pos_B)$\;
        }
        \textbf{return} $(false, pos_B)$\;
    }
\end{algorithm}

\begin{highlight}{Definition}{LimeGreen}
    The \textbf{\gls{intersection} search} consists in founding non-zero elements of two sparse arrays that have matching indices.
    This search is costly in software on classical CPUs, thus being key to improve performance of \acrshort{ip} algorithm.
\end{highlight}

\subsection{Outer-Product Algorithm}

In an \acrshort{op} algorithm, the inputs $A$ and $B$ (or $b$) are accessed sequentially, but the updates of the output $C$ (or $c$) require indirect accesses.
The pattern of the accesses, while the output is generated, is irregular and determined by the structure of the inputs.
Furthermore, if these \glspl{randUpdt} provoke cache \glspl{miss} where only one word within a cache-line is modified, the bandwidth to external memory is poorly utilized.
The \acrshort{op} algorithm generates several \acrfullpl{po} that must be accumulated to obtain the final output.
We can identify two strategies: \emph{immediate update} and \emph{deferred update}.
In the former, the final output is immediately updated as each \gls{psum} is generated.
If the output is stored using a \gls{spFormat}, either the index being updated does not yet exist, and the new \gls{psum} must be inserted.
Or, if it exists, a lookup must be done to locate it, the update performed and the result written back.
The problem associated with combining the \acrshortpl{po} is called \gls{reduction}.

In Algorithm~\ref{alg:bg:OPSpMSpMCSR}, we present two pseudo-code for an \acrfull{op} \acrshort{spmspm} using \emph{immediate update} (from line 1) and \emph{deferred update} (from line 7) with $A$ and $B$ stored in \acrshort{csc} and \acrshort{csr} formats, respectively.
The outer-loop ($K$, line 1 or 7) iterates over the columns of $A$ and the rows of $B$ (\gls{crank}).
The middle-loop (line 2 or 9) iterates over the non-zero elements of the current column ($k$) of $A$.
The inner-loop (line 4 or 11) iterates over the non-zero elements of the current row ($k$) of $B$.
Because we only iterate over the non-zero values, there is no need for an \gls{intersection} search.
The \gls{psum} is then computed (line 6 or 15).
Finally, in the \emph{immediate update} version, the \gls{psum} must be accumulated to $C$ (line 6).
These accesses are irregular, making it necessary to lookup the value in $C$, update it, and write it back.
This is a \gls{reduction} operation, performed on irregular data.

To avoid the complexities of these ``random'' \glspl{reduction} with \emph{immediate updates}, the \emph{deferred update} version postpones the accumulation and simply stores the \glspl{psum} in memory.
One such technique is called \emph{output batching}~\cite{Kiriansky2016_Milk}.
With this technique, arrays containing the indices and the \gls{psum} are streamed sequentially to memory ($\mathit{PO}_k$, lines 13-15).
Then, during a second pass (line 17), they are read back and the accumulations are performed, an approach which results in sequential memory accesses during both the first and second passes, with the downside that the volume of data transferred to memory is increased.
As we see in Chapter~\ref{cha:soa}, hardware accelerators may combine the \emph{immediate} and \emph{deferred update} techniques to optimize memory bandwidth.

\begin{algorithm}[htbp]

    %

    \KwData{$B(K, N)$ in \acrshort{csc}: $\mathit{B.val}, \mathit{B.idx}, \mathit{B.ptr}$}
    \KwData{$\mathit{PO}_k(M, N)$ for all $k \in \ldb 0, K-1 \rdb$ in \acrshort{coo}: $\mathit{PO_k.val}, \mathit{PO_k.row}, \mathit{PO_k.col}$}
    \KwData{$C(M, N)$ in \gls{dFormat} (for readability)}
    \KwResult{$C = A \times B$}
    \BlankLine
    \tcp{* * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * *}
    \tcp{* * * `Immediate updates' version * * * * * * * * * * * * * * * * * * * * * *}

            \tcp{Iterations over the non-zero elements of the k-th column of A}

            \For{$pos_B \in \ldb \mathit{B.ptr}_k, \mathit{B.ptr}_{k+1} \rdb$}{
                \tcp{Iterations over the non-zero elements of the k-th row of B}
                $j_B \gets \mathit{B.idx}_{pos_B}$\;
                \tcp{Column-index of the (k, pos\textsubscript{B}) non-zero element of B}
                $C_{i_A, j_B} \gets C_{i_A, j_B} + \mathit{A.val}_{pos_A} \times \mathit{B.val}_{pos_B}$\;
                \tcp{Reduction (to memory) of the partial sum into the (i\textsubscript{A}, j\textsubscript{B}) element of C (immediate update)}
            }
        }
    }
    \BlankLine

        \For{$pos_A \in \ldb \mathit{A.ptr}_k, \mathit{A.ptr}_{k+1} \rdb$}{
            \tcp{Iterations over the non-zero elements of the k-th column of A}
            $i_A \gets \mathit{A.idx}_{pos_A}$\tcp*[r]{Row-index of the (pos\textsubscript{A}, k) non-zero element of A}
            \For{$pos_B \in \ldb \mathit{B.ptr}_k, \mathit{B.ptr}_{k+1} \rdb$}{
                \tcp{Iterations over the non-zero elements of the k-th row of B}
                $j_B \gets \mathit{B.idx}_{pos_B}$\;
                \tcp{Column-index of the (k, pos\textsubscript{B}) non-zero element of B}
                $\mathit{PO_k.row}_{pos_\mathit{PO}} \gets i_A$\;
                $\mathit{PO_k.col}_{pos_\mathit{PO}} \gets j_B$\;
                $\mathit{PO_k.val}_{pos_\mathit{PO}} \gets \mathit{A.val}_{pos_A} \times \mathit{B.val}_{pos_B}$\; 
                \tcp{The k-th partial sum of the (i\textsubscript{A}, j\textsubscript{B}) element of C is stored in memory in the k-th partial output}
                $pos_\mathit{PO} \gets pos_\mathit{PO} + 1$\;

        }
        \tcp{Each partial output is generated sorted, row-wise}
    }

    \textbf{``Random'' \glspl{reduction}} occur when updating data from memory at irregular addresses, which happen on \emph{immediate update} \acrshort{op}.

The issues presented with \acrshort{ip} and \acrshort{op} can be generalized to two concepts: \gls{gather} and \gls{scatter}.
The \gls{gather} problem exists when the inputs are indirectly accessed, and need to be read from memory using \gls{metadata} to present the processor with the correct terms for computing the \glspl{psum}.
Typically, with \acrshort{ip}, the \gls{gather} problem is more difficult.
The \gls{scatter} problem exists when the order in which the output matrix/vector is generated is not sequential and is typically associated with the \acrshort{op} algorithm.
In the case of \gls{tiling} or with \acrshort{gust}, both \gls{gather} and \gls{scatter} must be performed creating a need for efficient \gls{intersection} searches and random \glspl{reduction}.

\begin{highlight}{Conclusion}{Red}
    There are \textbf{two main challenges} to address when computing sparse matrices.
    When reading irregular data from memory, addressing the \textbf{\gls{gather}} problem is key.
    It consists of processing efficient \textbf{\gls{intersection} searches}.
    When updating irregular data from memory, addressing the \textbf{\gls{scatter}} problem is key.
    It consists of \textbf{managing \acrshortpl{po}} in memory or processing efficient \textbf{``random'' \glspl{reduction}}.
    These two challenges are tied to \acrshort{ip} an \acrshort{op}, respectively, and are present in any higher \gls{rank} algorithms for sparse matrix computing.
\end{highlight}

\subsection{SpMSpM Algorithm Analysis}
\label{sec:bg:AnalyticalModel}

To better understand how the algorithms impact the number and types of memory accesses, we developed, and presented in Figure~\ref{fig:bg:AnalyticalModel_SpMSpM}, a model based on memory accesses counts for \acrshort{spmspm}.
We identify five memory access patterns: (1) those where a matrix is read only once, thus there is no opportunity for reuse, (2) where the accesses are sequential, but a \gls{fiber} is read multiple times, (3) where \glspl{fiber} are read consecutively, but the accesses within a \gls{fiber} are ``random'', (4) where the \glspl{fiber} are accessed ``randomly'', but within a \gls{fiber} the accesses are sequential, and (5) where the matrix elements are accessed multiple times, and with a ``random'' access pattern.

For each of the three algorithms, we counted the number of read and write accesses, based on the density of the input matrices, and the local memory storage available (``0D'', ``1D'' and ``2D''), as presented in Table~\ref{tab:bg:spmspm_access_count}.
We refer to a system with no local storage or cache as `0 Dimensional' (``0D''), a system which has local storage that is sufficient to hold a single row or column as `1 Dimensional' (``1D''), and a system which can cache an entire matrix as `2 Dimensional' (``2D'').
In Figure~\ref{fig:bg:AnalyticalModel_SpMSpM}, we plotted the number of memory accesses for \acrshort{spmspm}, in a logarithmic scale, based on a square matrix with $M = 10,000$.
Indeed, the number of non-zero elements in $C$ depends on the structure of the input matrices and in this model, we have assumed the non-zero entries in the input matrices are uniformly, randomly distributed.
This corresponds to a \emph{worst case} for sparse matrices (no spatial and temporal locality).
If $A$ or $B$ are \gls{banded} matrices, for example, there would be fewer non-zero elements in $C$.
In each graph, we have separated the accesses into $A$ (\textit{bottom}), $B$ (\textit{middle}) and $C$ (\textit{top}).
The color code shows the access pattern, based on the five categories described above.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{Chapter_Background/Figures/PDF/RawSpMSpMAccessesCountAnalysis.pdf}
    \caption{Model of Memory Access Counts Depending on Algorithm and Local Memory Size}
    \label{fig:bg:AnalyticalModel_SpMSpM}
\end{figure}

In the top-row of Figure~\ref{fig:bg:AnalyticalModel_SpMSpM} (graphs~\ding{192}, \ding{193} and~\ding{194}), we start by considering a na\"ive system which has no local storage (``0D''), thus all data comes from main memory.
From the graphs in this row, we clearly see that \acrshort{ip} requires more accesses at low density, as the $A$-matrix must be read $N$ times (graph~\ding{192}).
Whereas for \acrshort{op} and \acrshort{gust}, the $A$-matrix is read only once (thus the \textit{grey} color).
For \acrshort{op} and \acrshort{gust} the total number of accesses is the same, however, with \acrshort{op} (graph~\ding{193}), the accesses to $C$ are ``random'' (\textit{red}).
With \acrshort{gust} (graph~\ding{194}), the accesses to $B$ are sequential within the rows (\textit{orange}) and the $C$-rows are generated sequentially (\textit{yellow}), thus \acrshort{gust} avoids having any fully ``random'' accesses (\textit{red}).

Since main memory accesses are the principal bottleneck, systems integrate local storage (such as buffers or caches).
Thus, we consider the case of a system with sufficient local memory to store one full row, per matrix (equivalent to few sparse rows, ``1D'').
We assume this local storage is perfect (no \glspl{miss}) and we re-calculate the number of remaining main memory accesses, as shown in the middle-row of Figure~\ref{fig:bg:AnalyticalModel_SpMSpM} (graphs~\ding{195}, \ding{196} and~\ding{197}).
The matrices are assumed to be initially stored in main memory, thus they must be accessed at least once.
As can be seen, for \acrshort{ip} (graph~\ding{195}), the number of $A$-accesses is reduced, because rows are reused, thus the total number of $A$-accesses becomes equal to that in \acrshort{op} and \acrshort{gust}.
For \acrshort{op} (graph~\ding{196}), the single row buffer greatly reduces the number of $B$-accesses, however, it does nothing for $C$ (despite the appearance, due to the logarithmic scale).
For \acrshort{gust} (graph~\ding{197}), the single row buffer is sufficient to cache the \gls{reduction} operations in $C$, and thus, it now has fewer overall accesses than \acrshort{op}.
It is also interesting to note that \acrshort{op} and \acrshort{gust} scale better than \acrshort{ip} with lower densities, as seen by the shallower slope.

Finally, we have considered the extreme case where an system has a buffer of sufficient size to store two sparse matrices (``2D''), which is shown in the bottom-row of Figure~\ref{fig:bg:AnalyticalModel_SpMSpM} (graphs~\ding{198}, \ding{199} and~\ding{200}).
Although this case is not practical for large matrices, through the proper use of \gls{tiling} to locally reduce the dimensions of a sub-matrix, this case can be achieved at the level of a \gls{tile}.
In this case, the total number of external accesses is equal for the three algorithms, as each matrix is accessed only once from main memory.
For \acrshort{ip} (graph~\ding{198}), it is interesting to note that such a large buffer is needed to locally address the \gls{intersection} search in $B$.
Similarly for \acrshort{op} (graph~\ding{199}), a large storage is required to address the \glspl{reduction} in $C$.
For \acrshort{gust}, in fact, it is not necessary to buffer the entire $B$-matrix.
Indeed, buffering a few rows (corresponding to $nnz_r(A)$) is sufficient to achieve the number of accesses shown in graph~\ding{200}.
This corresponds to an intermediate design point (multiple row buffers for $B$) not explicitly shown in the figure (between graphs~\ding{197} and~\ding{200}).

When viewed overall, Figure~\ref{fig:bg:AnalyticalModel_SpMSpM} highlights the benefits of \acrshort{gust}.
We clearly see that \acrshort{gust} does not require any fully ``random'' accesses (\textit{red}).
With a single row buffer, it is possible to address the \glspl{reduction} in $C$.
And, with a limited number of row buffers, the number of external memory accesses for $B$ can be reduced.
Based on this analysis, it is clear that, among the three base algorithms, \acrshort{gust} is the best candidate for hardware implementation of \acrshort{spmspm}.

\begin{highlight}{Note}{SkyBlue}
    For \acrshort{spmv}, such an analysis does not show significant differences between \acrshort{ip} and \acrshort{op}, as both algorithms require similar number of memory accesses.
    However, \acrshort{op} shows more potential for parallelism and data-reuse as seen in the previous section (\ref{sec:bg:AlgosHWComp}), thus being preferred for \acrshort{spmv} hardware accelerators.
\end{highlight}

\begin{highlight}{Conclusion}{Red}
    Based on a high level analysis of the memory accesses on \acrshort{spmspm}, \textbf{\acrlong{gust}'s algorithm} appears as the \textbf{best trade-off} to minimize memory traffic with reasonable amount of local storage.
    The second best option is \acrshort{op}, which requires fewer memory accesses than \acrshort{ip} when the matrices have a low density.
\end{highlight}
