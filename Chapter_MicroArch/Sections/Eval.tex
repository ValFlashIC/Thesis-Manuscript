\section{Evaluation}
\label{sec:arch:eval}

\lettrine{I}{n} this section, we aim to evaluate the performance of various cache configurations and their impact on memory traffic and latency during \acrshort{rtl} simulations.
We will first outline the methodology used for these evaluations, followed by a detailed analysis of the results obtained from the simulations, ending with comparisons with the previous models.

% %%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\subsection{Methodology for RTL Simulations}

The starting point is a CVA6~\cite{Zaruba2019_CVA6} RISC-V core with HPDcache~\cite{Fuguet2023_HPDcache} as an L1 data-cache with size 32~KiB ($S=128$, $W=4$, $l=8$ elements).
This serves as our baseline for performance comparison which we refer to as the \acrfull{gc}.
Next we evaluate a \gls{redC} which we derived from the HPDcache and configured to have two regions: a generic region with 16~KiB, and a \gls{reduction} region with 16~KiB whose behavior is the same as the \acrshort{drc} region of \acrlong{spdc}.
We refer to this design as \acrshort{redc}.
Finally, we evaluate \acrfull{spdc} with a reference configuration of: \acrshort{grc} = 8~KiB, \acrshort{drc} = 16~KiB, \acrshort{srt} = 8~KiB, which corresponds to the 25/50/25\% configuration selected in Chapter~\ref{cha:spdc}.
The configurations are summarized in Table~\ref{tab:arch:region_configs}.

All our simulations are performed using Siemens Questasim\texttrademark~2023.3 and the memory accesses outside the L1 incur a fixed latency of 64 clock cycles.
We do not aim to evaluate a full system in this work and thus use a single-core, single-cache-level environment.
The CVA6 core is single-issue and can process at most one instruction per cycle.
The fixed memory latency of 64 cycles provides a reasonable approximation of the latency incurred by requests passing through the other memory levels, and a deterministic memory model makes the results easily reproducible.

Currently, our SystemVerilog implementation only supports 64-bit integer additions, precluding floating-point accumulations.
Extending \acrlong{spdc} to support floating-point operations remains future work.
We employ 64-bit integers throughout and assume this is representative because sparse kernels are memory-bound, and 64-bit integers have the same data size as FP64 doubles.

A second implementation constraint is that region sizes for \acrshort{grc}, \acrshort{drc}, and \acrshort{srt} must be powers of two in terms of \glspl{set}, unlike the analytical and Python models described in Chapters~\ref{cha:spdc} and~\ref{cha:theory}.
This limitation motivates our choice of a 50/50/0\% configuration for \acrshort{redc}, which we expect to slightly underperform compared to a configuration which allocates more space to the \acrshort{drc}.
Future optimizations of the implementation could relax this constraint.

\begin{highlight}{Conclusion}{Red}
    The \acrshort{rtl} simulation evaluation was performed using three cache configurations, \acrshort{gc}, \acrshort{redc}, and \acrlong{spdc}, using a single-core CVA6 processor with a single cache level.
\end{highlight}

\subsection{Sparse Formats Used for the RTL Simulations}
\label{sec:arch:bacsc}

The \acrshort{csr}/\acrshort{csc} formats are widely used in high-performance computing.
As we are using an \acrshort{op} algorithm, we use the \acrshort{csc} format for the majority of our simulations.
However, in our evaluation with \gls{banded} matrices, we also simulated \acrlong{spdc} with a modified \acrshort{csc} format tailored for \gls{banded} matrices which we call \acrshort{bacsc}.
Indeed, since \acrlong{spdc} must know where the \gls{banded} region lies, and to avoid calculating it using precious processor instructions, this information can be placed in memory with the matrix.
As shown in Figure~\ref{fig:arch:BaCSC}, compared to a \acrshort{csc}, we added extra information in the $ptr$ table to identify the beginning and end of the \gls{band}.
Indeed, three pointers, instead of one, are now needed per column, as shown with the example of column 2 in \textit{yellow}.
The column is split in one in-\gls{band} region in the middle (IB, from $ptr$ 7 to 9) and two out-of-\gls{band} regions around (OB, from $ptr$ 6 to 7, and 9 to 10).
The $ptr$ table size thus becomes $3 \times M + 1$.
With this format, we can avoid the computational cost of identifying in which region non-zero elements are, while traversing the matrix, with the drawback of having to preprocess the matrix to store this additional information.

    \caption{Example of a Sparse Matrix in \acrshort{bacsc} Format}

\end{figure}

\begin{highlight}{Conclusion}{Red}
    We employ the \acrshort{csc} format for most simulations and additionally evaluate \acrlong{spdc} using \acrshort{bacsc}, a specialized format for \acrlong{spdc} which specifies the location of the \gls{banded} region of the matrix.
\end{highlight}

\subsection{Software Implementation of Outer-Product SpMV Kernel}
\label{sec:arch:sw}

The code running on the CVA6 is written in generic C, compiled with \textit{gcc} 11.2.0 and -O3 optimization.
The \acrshort{red}-related instructions used in the \acrshort{op} \acrshort{spmv} kernel are manually inserted using \textit{asm} directives, as illustrated in Source Code~\ref{src:arch:op_spmv}.

\sourcecode{Chapter_MicroArch/Code/op_spmv.c}{C Code for the \acrshort{op} \acrshort{spmv} with \acrshort{red} Operations}{src:arch:op_spmv}{htbp}

The code shown is for \acrlong{spdc}.
Variants for other designs differ as follows: \acrshort{redc} omits region selection (line~15) and invokes \acrshort{reg} once before the loops with \acrshort{ibr} as the argument; \acrshort{gc} requires no \acrshort{reg} instruction and replaces \acrshort{red} with standard accumulation: $c[idx] \acc val$.

For \acrlong{spdc}, it is necessary to know the target region for each \acrshort{red} instruction based on the size of the \gls{band} (line~15).
The half-\gls{band} width $w_\mathit{hB}$ is derived from Chapter~\ref{cha:theory} (Equation~\ref{eq:theory:in_band_cond}) to ensure the in-\gls{band} region fits within the \acrshort{drc} independent of matrix structure.
In our reference configuration ($s_C(\mathit{DRC}) = 2048$, $l = 8$), $w_\mathit{hB} = 1021$.
Region selection follows Equation~\ref{eq:arch:reg_select} via an inline function, avoiding conditional branches within the inner-loop.

\begin{equation}
    \label{eq:arch:reg_select}
    \forall (i,j) \in \ldb 0, M-1 \rdb \times \ldb 0, N-1 \rdb, \; \mathit{region} = \left\{
    \begin{tabular}{ll}
        \acrshort{ibr} & if \; $|i-j| < w_\mathit{hB}$ \\
        \acrshort{obr} & else
    \end{tabular} \right.
\end{equation}

On-the-fly region computation introduces overhead in the critical inner-loop.
To mitigate this, we propose an alternative implementation (Source Code~\ref{src:arch:op_spmv_bacsc}) that precomputes regions and stores them using the \acrshort{bacsc} format (see Section~\ref{sec:arch:bacsc}).
Each column iteration begins with lookups in the outer-loop (line~5) to retrieve pointers for three sections: top out-of-\gls{band} (\textit{OB1}), in-\gls{band} (\textit{IB}), and bottom out-of-\gls{band} (\textit{OB2}).
The column is then processed through three separate inner-loops (lines~13, 21, 29) without region computation overhead.

\sourcecode{Chapter_MicroArch/Code/op_spmv_bacsc.c}{Optimized C Code for the \acrshort{op} \acrshort{spmv} when Using \acrshort{bacsc}}{src:arch:op_spmv_bacsc}{htbp}

Each approach trades off one limitation for another.
\acrlong{spdc}(\acrshort{csc}) uses simple \acrshort{csc} format with minimal preprocessing and reduced memory overhead, but incurs additional inner-loop instructions and thus latency per iteration.
\acrlong{spdc}(\acrshort{bacsc}) reduces instructions and latency at the cost of matrix preprocessing into \acrshort{bacsc} format and an increased memory footprint for the matrix.
Both variants are evaluated in the results section.

These software approaches avoid hardware modifications, providing flexibility.
However, hardware-computed regions with \acrshort{csc} format would eliminate the noted drawbacks.
A detailed breakdown of each design's instruction characteristics:
\begin{itemize}
    \item \acrshort{gc}: 10 inner-loop instructions; 3 instructions per \acrshort{red} with critical load dependency.
    \item \acrshort{redc}: 8 inner-loop instructions; 1 ``fire-and-forget'' \acrshort{red} instruction; 1 \acrshort{reg} setup per entire computation.
    \item \acrlong{spdc}(\acrshort{csc}): 14 inner-loop instructions; 1 ``fire-and-forget'' \acrshort{red} instruction; region computation within inner-loop.
    \item \acrlong{spdc}(\acrshort{bacsc}): 8 inner-loop instructions; 1 ``fire-and-forget'' \acrshort{red} instruction; outer-loop overhead.
\end{itemize}

To prevent software from limiting performance, we unroll inner-loops by a factor of 8 (not shown in Source Codes~\ref{src:arch:op_spmv} and~\ref{src:arch:op_spmv_bacsc}).
Each unrolled iteration executes 8 logical iterations with optimized instruction scheduling.
With cache-line size $l = 8$, this unrolling hides read-\gls{miss} latency through \glspl{hit} on consecutive addresses.
``Random'' \gls{reduction} accesses do not benefit from this latency hiding.
Consequently, \acrshort{gc} experiences potential latency from both \glspl{reduction} reads and writes, while \acrshort{redc} and \acrlong{spdc} benefit from ``fire-and-forget'' semantics and consecutive \gls{reduction} scheduling.

\acrlong{spdc}(\acrshort{csc}) is constrained to 4\tm unrolling due to an insufficient number of register in the CVA6 CPU.
This limitation, addressable with a higher-register CPU, highlights another advantage of hardware based region computation.

\begin{highlight}{Conclusion}{Red}
    We utilize four distinct software implementations of the \acrshort{op} \acrshort{spmv}, each customized for our four simulations and highly unrolled to minimize performance limitations due to loop overheads.
\end{highlight}

\subsection{Matrix Selection for the RTL Evaluation}

We have selected an extended set of 62 square matrices from the SuiteSparse Matrix Collection~\cite{Kolodziej2019_SuiteSparse} which includes virtually all the matrices used by recent hardware accelerators~\cite{Zhang2021_Gamma, Xie2021_SpaceA, Zhang2022_ASA, Srikanth2021_SortCache, Pal2018_OuterSPACE} (full list in Appendix~\ref{app:mat_list}).
We verified that this set is diverse containing \gls{banded} and non-\gls{banded} sparse matrices with various sizes ($M \in [1.2\times10^{2}, 2.0\times10^{6}]$), numbers of non-zero elements ($nnz \in [2.2\times10^{3}, 1.4\times10^{7}]$) and a wide range of densities ($d \in [1.4\times10^{-6}, 7.8\times10^{-1}]$).

Based on our experimental results presented in the following section, we categorize these matrices into two groups according to their performance characteristics.
To facilitate this classification, we introduce a new metric, $d_\mathit{cd}$, defined as the average density of cache-line-sized clusters within matrix columns.
Specifically, for a cache with line size $l$, this metric quantifies the average density of non-empty cache-lines (or vertical-stripes as referenced in Chapter~\ref{cha:theory}), ranging between $\frac{1}{l}$ and 1.

This metric provides a fine grain view of matrix density, which aligns more closely with cache characteristics.
When $d_\mathit{cd}$ approaches $\frac{1}{l}$ for a given matrix, it indicates that most non-zero elements are separated by at least $l-1$ zeros.
Consequently, the cache-lines are often nearly empty, which may render the line-wise \acrshort{drc} of a \acrshort{redc} inefficient, while the element-wise \acrshort{srt} of \acrlong{spdc} becomes more beneficial.
Conversely, as $d_\mathit{cd}$ increases, the non-zero elements of the matrix cluster more closely, enhancing the efficiency of the \acrshort{drc}.

In our experiments with $l = 8$, we establish two groups of matrices based on a $d_\mathit{cd}$ threshold of 20\%.
This threshold corresponds to either one to two non-zero elements per eight.
At this point, this threshold may appear somewhat arbitrary and it, of course, depends on the processor and caches.
As we presents the results, this choice, for our CVA6 core, will become clearer.

Using the threshold based on $d_\mathit{cd}$, we categorize 21 matrices in group~1 ($d_\mathit{cd} \leqslant 20\%$) and 41 in group~2 ($d_\mathit{cd} > 20\%$).
We have selected 13 well-known matrices from both groups that are frequently used for performance evaluation, as illustrated in Table~\ref{tab:arch:matrices}.
The selected matrices represent a range of overall matrix densities ($d$) and $d_\mathit{cd}$ values, as depicted by the \textit{red} and \textit{purple} points in the scatter plot in Figure~\ref{fig:arch:matrices}.
In the following evaluation, we report the performance results of the 13 matrices from group~1 and~2, the geometric means for both groups (covering 21 matrices in group~1 and 41 in group~2), and the geometric mean for the full set of 62 matrices.

\subsection{Results Analysis}

In this section, we present and analyze performance results in two complementary ways.
First, in Sections~\ref{sec:arch:perfMT} and~\ref{sec:arch:perfSpeedup}, we report memory traffic and speedup for \acrshort{redc} and \acrlong{spdc} normalized relative to the baseline \acrshort{gc}.
Second, in Section~\ref{sec:arch:perfGroups}, we report memory traffic and speedup for \acrshort{gc}, \acrshort{redc}, and \acrlong{spdc} normalized per matrix non-zero elements ($\mathit{nnz}$).
This two-part analysis provides complete understanding of performance behavior across the two matrix groups defined in the previous section.

% %%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\subsubsection{Memory Traffic Performance}
\label{sec:arch:perfMT}

For all benchmark matrices, we simulated the \acrshort{rtl} model executing the full \acrshort{op} \acrshort{spmv} and measured output memory traffic and input memory traffic.
In Figures~\ref{fig:arch:outMT} and~\ref{fig:arch:totMT}, we show the output memory traffic and the total memory traffic (output + input), respectively, for \acrshort{redc}, \acrlong{spdc} with \acrshort{csc}, and \acrlong{spdc} using the \acrshort{bacsc} format, normalized compared to the output traffic when simulating with \acrshort{gc}.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{Chapter_MicroArch/Figures/PDF/OutMemTrafficFromRTL.pdf}
    \caption{Output Memory Traffic Normalized Compared to \acrshort{gc}, Measured from \acrshort{rtl} Simulations}
    \label{fig:arch:outMT}
\end{figure}

As shown in Figure~\ref{fig:arch:outMT}, looking at the geometric mean on all 62 matrices, our main result is that \acrlong{spdc} achieves a 3.9\tm output memory traffic reduction compared to \acrshort{gc}, which is 2.8\tm better than \acrshort{redc} (1.4\tm).
While \acrlong{spdc} performs consistently for both groups of matrices, \acrshort{redc} lacks hardware adaptation for \gls{irrAccess} patterns, making its performance heavily group-dependent.
For low $d_{\mathit{cd}}$ (sparse cache-lines, group~1), the line-wise cache proves ineffective.
Conversely to \acrshort{redc}, \acrlong{spdc} maintains dense data in the \acrshort{drc} across columns, enabling the rotating pattern from Chapter~\ref{cha:theory}.
Outside the \acrshort{drc}, the element-wise \acrshort{srt} organization efficiently avoids transferring mostly empty cache-lines.

For group~1 matrices, \acrshort{redc} traffic exceeds \acrshort{gc} because \acrshort{gc}'s cache is 2\tm larger than \acrshort{redc}'s \acrshort{drc}, reducing performance\footnote{Recall that for the benchmarking, the total cache size was kept constant at 32~KiB. The \acrshort{redc} has a 16~KiB generic region and a 16~KiB \acrshort{drc}, whereas the \acrshort{gc} can flexibly use the full 32~KiB.}.
For group~2 matrices, higher cache-line density compensates, allowing \acrshort{redc} to leverage near-memory accumulation through increased \gls{hit}-rates.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{Chapter_MicroArch/Figures/PDF/TotMemTrafficFromRTL.pdf}
    \caption{Total Memory Traffic for Input Matrix $A$, Input Vector $b$ and Output Vector $c$, Normalized Compared to \acrshort{gc}, Measured from \acrshort{rtl} Simulations}
    \label{fig:arch:totMT}
\end{figure}

In Figure~\ref{fig:arch:totMT} we show the total memory traffic broken down by input data reads ($A$ and $b$, \textit{hatched}) and output vector updates ($c$, \textit{plain}).
As the two \acrlong{spdc} variants show no difference in output traffic, the total memory traffic shows that the increased matrix storage, which incurs additional memory reads, for \acrlong{spdc} with \acrshort{bacsc} has an overhead below 10\% (1.6\tm versus 1.5\tm when looking at the geometric mean on all 62 matrices).
Otherwise, input memory traffic ($A$ and $b$, \textit{hatched}) is constant on every configurations and both groups of matrices\footnote{\textit{Hatched} bars seem to vary between matrix groups. It is a visual effect of the normalization versus \acrshort{gc}.}.

\begin{highlight}{Conclusion}{Red}
    \textbf{\acrlong{spdc} achieves 3.9\tm output memory traffic reduction versus \acrshort{gc}}, substantially outperforming \acrshort{redc}'s 1.4\tm reduction.
    Group~1 matrices (low $d_\mathit{cd}$) benefit most from traffic reduction due to sparse cache-lines, while group~2 (high $d_\mathit{cd}$) shows more modest gains as denser cache-lines are inherently more efficient.
    The total memory traffic overhead for the \acrshort{bacsc} format is under 10\%.
\end{highlight}

% %%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\subsubsection{Latency Performance}
\label{sec:arch:perfSpeedup}

We now analyze total compute time, recalling that all main memory reads incur a fixed 64 cycle latency.
Figure~\ref{fig:arch:speedup} presents the speedup relative to \acrshort{gc}.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{Chapter_MicroArch/Figures/PDF/Speedups.pdf}
    \caption{Speedup Compared to \acrshort{gc}, Measured from \acrshort{rtl} Simulations}
    \label{fig:arch:speedup}
\end{figure}

Group~1 matrices show speedup across all cache configurations, whereas group~2 shows slowdown.
Overall, looking at the geometric mean on all 62 matrices, \acrshort{redc} achieves no speedup over \acrshort{gc}, \acrlong{spdc} with \acrshort{csc} incurs 1.2\tm slowdown, and \acrlong{spdc} with \acrshort{bacsc} achieves 1.1\tm speedup.

For group~1 matrices, \acrshort{redc} obtains a small speedup (1.1\tm) because \acrshort{red} instructions are ``fire-and-forget'', replacing multiple read-accumulate-write operations.
However, blocking occurs when the \textit{accumulation table} processes prior \glspl{eviction} limiting the benefit.
Group~1's sparse cache-lines trigger frequent \gls{eviction} bursts, dominating the performance and making the system highly memory-bound.
Thus, \acrlong{spdc}'s substantial memory traffic reduction translates directly to speedup by eliminating costly line-wise \glspl{eviction}.
\acrlong{spdc} with \acrshort{bacsc} performs best, with a 1.2\tm speedup, by eliminating region calculation overhead in the critical loop.

For group~2 matrices, our single-core RISC-V processor becomes compute-bound.
Higher cache-line density means \glspl{eviction} are typically followed by non-blocking \glspl{hit}.
In the extreme case, 7 \glspl{hit} following an \gls{eviction} require 56-98 instructions (8-14 instructions per iteration, see Section~\ref{sec:arch:sw}) before the next potential \gls{eviction}, sufficient to hide the 64 cycle memory latency.

Consequently, \acrlong{spdc} with \acrshort{bacsc} surpasses \acrshort{gc} performance.
\acrshort{redc} underperforms by 1.1\tm due to its 2\tm smaller \acrshort{drc} versus \acrshort{gc}.
With a larger \acrshort{drc}, \acrshort{redc} would yield comparable performance.
\acrlong{spdc} with \acrshort{csc} underperforms by 1.4\tm because, in compute-bound regimes, reducing inner-loop instruction count is critical for performance.

\begin{highlight}{Conclusion}{Red}
    For group~1 matrices (low $d_\mathit{cd}$), \textbf{\acrlong{spdc}(\acrshort{bacsc}) performs best}, with a 1.2\tm speedup, by eliminating costly \glspl{eviction} and region computation overhead.
    For group~2 matrices (high $d_\mathit{cd}$), the workload becomes compute-bound but \acrlong{spdc}(\acrshort{bacsc}) still matches the \acrshort{gc} performance.
    Overall, \acrlong{spdc}(\acrshort{bacsc}) achieves a 1.1\tm average speedup, while the other configurations underperform on average, with \textbf{gains concentrated in memory-bound scenarios}.
\end{highlight}

\subsubsection{Further Discussion Regarding the Two Performance Groups}
\label{sec:arch:perfGroups}

In the two previous sections, we showed performance normalized to \acrshort{gc}, highlighting how \acrlong{spdc} affects baseline performance, assuming it was used as an alternative cache.
Here, we further analyze performance by normalizing against the number of non-zero elements ($\mathit{nnz}$), yielding average raw output memory traffic and latency per \acrshort{op} \acrshort{spmv} iteration, or equivalently per \gls{reduction} processed.
This reveals raw performance variations across matrix groups and configurations (\acrshort{gc}, \acrshort{redc}, \acrlong{spdc}(\acrshort{csc}), and \acrlong{spdc}(\acrshort{bacsc})).
We can thus analyze performance behaviors across the two matrix groups without the normalization bias introduced by comparing against \acrshort{gc}.
Figure~\ref{fig:arch:perfOnMat} (eight graphs: four rows, two columns) organizes results as follows: each row of graphs represents a configuration; columns show output memory traffic and latency normalized by $\mathit{nnz}$.
Within each graph, dots represent the 62 matrices scattered by density ($d$) and $d_\mathit{cd}$, and \textit{white}-to-\textit{red} coloring indicates performance data (output traffic or latency) which values are visible on the right-side bars.
For both output traffic and latency metrics, for which lower values are better, white dots indicates matrices achieving better performance (\textit{white} and \textit{red} corresponds to low and high values, respectively).
Moreover, each graph is annotated with the geometric mean performance over the group~1 matrices (G1, before the \textit{dashed}, vertical threshold at 20\%), the geometric mean performance over the group~2 matrices (G2, after the \textit{dashed}, vertical threshold at 20\%), and the performance of group~1 compared to group~2 (G1/G2, higher it is, better performance group~2 has compared to group~1).
Note that the performance scale is not the same across graphs (varying minimums and maximums on the right-side bars), as we no more directly compare configurations between each other (previous sections), but solely compare performance between matrices for each configuration.

\begin{figure}[htbp]
    \centering
    \begin{subfigure}{0.49\linewidth}
        \centering

We first notice that \acrshort{gc} (graphs~\ref{fig:arch:perfOnMat:00} and~\ref{fig:arch:perfOnMat:01}) shows group-dependent variation.
Indeed, the \acrshort{gc} output memory traffic exhibits 17.6\tm higher traffic on group~1 than group~2 (G1/G2).
This normalization bias relative to \acrshort{gc} becomes evident when examining raw per-$\mathit{nnz}$ metrics, explaining why \acrlong{spdc} demonstrates greater relative benefit to group~1 matrices in previous comparisons against \acrshort{gc} (Figure~\ref{fig:arch:outMT}).

Input memory traffic exhibits the same bias across groups.
Notably, \acrshort{gc} input memory traffic normalized per $\mathit{nnz}$ is only 1.1\tm higher for group~1 compared to group~2.
Consequently, when examining the total memory traffic in Figure~\ref{fig:arch:totMT} from Section~\ref{sec:arch:perfMT}, the input traffic component (\textit{hatched} bars) remains nearly constant between groups.
Output traffic (\textit{plain} bars) thus dominates \acrshort{gc} performance in group~1, representing 75\% of the total, whereas it accounts for less than 25\% in group~2.
This observation supports that the output traffic reduction achieved by \acrlong{spdc} provides substantially greater benefit to group~1 matrices, where output traffic is the primary bottleneck, compared to group~2 matrices, which already exhibit low output traffic under \acrshort{gc}.

Going back on the graphs, we then observe that group~2 consistently outperforms group~1 across all configurations and metrics (all eight graphs), \eg the \textit{red} dots are nearly all group~2 matrices, on the left side.
Group~1 exhibits from 8.27\tm to 31.4\tm higher memory traffic per $\mathit{nnz}$ (G1/G2s of first column of graphs) and from 1.20\tm to 1.86\tm higher latency per $\mathit{nnz}$ (G1/G2s of second column of graphs) than group~2.
When $d_\mathit{cd}$ is low (left of each graph), cache-lines remain nearly empty, increasing \gls{eviction} frequency.
Additionally, low $d_\mathit{cd}$ causes consecutive \acrshort{op} \acrshort{spmv} iterations to generate \gls{eviction} bursts.
Consequently, when $d_\mathit{cd}$ is low: (1) \gls{eviction} count increases and (2) \glspl{eviction} cluster temporally.
The first effect raises output vector $c$ traffic, potentially triggering critical memory reads that affect latency.
The second exacerbates latency: when consecutive iterations process \glspl{eviction}, the intervening $\sim$10 instructions cannot hide our fixed, 64 cycle read latency (insufficient loop unrolling).
As visible in graph~\ref{fig:arch:perfOnMat:01}, worst-case group~1 matrices average 64 cycles latency on \acrshort{gc} (maximum \textit{red} value on the right-side bar of the graph).
Repeatedly waiting for \glspl{eviction} causes processor stalls when $d_\mathit{cd}$ is low, making the performance highly memory-bound.

As $d_\mathit{cd}$ increases (right of each graph), consecutive iterations exhibit cache-line \glspl{hit}.
\Gls{eviction} frequency and burst clustering both decrease.
Between successive \glspl{eviction}, \gls{hit} iterations execute sufficient instructions to overlap with prior read latency.
Once the \gls{eviction} latency falls below the loop iteration time, the workload becomes compute-bound.

Group~1 benefits significantly from memory traffic reduction, while group~2 is constrained by the latency of the inner-loop in software implementation.
Previous sections demonstrated this while taking \acrshort{gc} as the performance reference.
While \acrshort{gc} and \acrshort{redc} remain bottlenecked, \acrlong{spdc} substantially reduces costly \glspl{eviction} and avoids bursting by accumulating with high \gls{hit}-rates in the \acrshort{drc} or evicting via low-cost, element-wise transactions within the \acrshort{srt}.
Consequently, \acrlong{spdc} delivers significant latency reduction for the matrices in group~1 which are memory-bound.
For the compute-bound matrices in group~2, traffic reduction yields modest latency improvements.
Here, the \acrshort{bacsc} format reduces the number of instructions in the inner-loop by avoiding the region calculation.

For \acrshort{gc}, loop unrolling allows the \textit{miss handler} to queue up to eight concurrent \gls{eviction} reads through its multiple entries.
However, this proves insufficient for group~1 matrices, as the unrolled inner-loop contains too many instructions to hide the memory latency.
Additionally, the \textit{miss handler}'s parallelism cannot be fully exploited if the \textit{accumulation table} lacks sufficient entries to process the corresponding \acrshort{red} \glspl{eviction}.
Increasing the \textit{accumulation table} to eight entries similarly yields negligible speedup (<1\%, not shown in any figure) for \acrshort{redc} and \acrlong{spdc}, since the bottleneck remains the critical path length of the inner-loop instructions.

\begin{highlight}{Conclusion}{Red}
    Overall, \textbf{group~2 always outperforms group~1} when compared based on raw memory traffic and latency per $\mathit{nnz}$.
    Group~1 matrices (low $d_{\mathit{cd}}$) are \textbf{memory-bound} due to sparse cache-lines causing frequent \gls{eviction} bursts, while group~2 matrices (high $d_{\mathit{cd}}$) are \textbf{compute-bound} with dense cache-lines hiding \gls{eviction} latency.
    Consequently, \acrlong{spdc} achieves speedup in group~1 through memory traffic reduction, but only matches baseline performance in group~2 where the inner-loop latency becomes the bottleneck.
\end{highlight}

\subsubsection{Area Overhead}

Both HPDcache and \acrlong{spdc} systems were synthesized in GF22FDX technology (22 nm) using Synopsis DC\texttrademark.
The original area was 0.99~mm\textsuperscript{2} and the \acrlong{spdc} version was 1.3~mm\textsuperscript{2}, an increase of 31\%.
These results are preliminary since the \acrlong{spdc} \acrshort{rtl} implementation has not undergone synthesis optimization.
This shows that \acrlong{spdc} could reasonably be integrated into a processor, providing a large performance improvement (3.9\tm in memory traffic), with an area overhead that is much lower than predicted by Pollack's law~\cite{Borkar2007_AreaOverheads} on a single core ($< 3.9^2 = 1500\%$).
For context, similar state-of-the-art accelerators scaled to 22 nm technology for a single cache exhibit the following area overheads: PHI requires 9\%, SortCache requires 172\%, ASA requires <1\% and C\textsuperscript{3}ache requires 9\%.
In comparison, \textit{standalone} accelerators have area overheads about one order of magnitude higher at least.

\subsection{Comparison with the Analytical Model}

To validate consistency between our models and implementations, we compare the normalized output memory traffic produced by three sources: the analytical model of Chapter~\ref{cha:theory}, the Python model of Chapter~\ref{cha:spdc}, and the \acrshort{rtl} implementation presented in this chapter.
For a fair comparison, the caches for \acrshort{redc} and \acrlong{spdc} are configured to 32~KiB total (see Table~\ref{tab:arch:region_configs}): 50/50/0\% for \acrshort{redc} and 25/50/25\% for \acrlong{spdc}.
The analytical model accepts global parameters and density distributions rather than actual matrices, so we synthesize \gls{banded} matrices for the Python and \acrshort{rtl} simulations that match those parameters.
All synthetic matrices use $M = 10^{4}$, half-\gls{band} width $w_\mathit{hB} = 1021$, in-\gls{band} density $d_\mathit{IB} = 10^{-2}$, and out-of-\gls{band} density $d_\mathit{OB}$ varying from $10^{-7}$ to $10^{-2}$.
Note that at $d_\mathit{OB} \simeq 8\times10^{-3}$, approximately one non-zero element exists per column in the out-of-\gls{band} region, whereas at $d_\mathit{OB} \simeq 8\times10^{-7}$, approximately one non-zero element exists in the entire out-of-\gls{band} region of the full matrix, resulting in a nearly \gls{PBanded} matrix.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{Chapter_MicroArch/Figures/PDF/graph11_25.12.02.pdf}
    \caption{Normalized Output Memory Traffic Depending on the Out-of-Band Density for the \acrshort{redc} and \acrlong{spdc} Configurations, Through Different Simulation Models}
    \label{fig:arch:graph11}
\end{figure}

Figure~\ref{fig:arch:graph11} shows the results: \acrlong{spdc} (\textit{green}) outperforms \acrshort{redc} (\textit{red}) in normalized output memory traffic ($\mathit{normMT}$).
Two observations are noteworthy.
First, the three approaches agree most closely when $d_\mathit{OB}$ is high.
Indeed, at $d_\mathit{OB} = 10^{-2}$ there is no measurable difference for \acrshort{redc}, while \acrlong{spdc} exhibits a 0.23 difference in $\mathit{normMT}$.
At the smallest $d_\mathit{OB}$ the discrepancies reach 0.58 (0.55 + 0.03) element transferred in memory per \gls{reduction}.
Overall trends hold across models and deviations remain small.

Second, the analytical model consistently overestimates traffic (except on two points), the Python model consistently underestimates it, and the \acrshort{rtl} results fall between the two.
Indeed, the analytical model operates on matrix distributions rather than actual generated matrices, using a uniform distribution in this case.
At very low densities ($\leqslant$ 10\textsuperscript{-2}), the analytical model exhibits consistently poor cache \gls{hit}-rates.
In contrast, the Python and \acrshort{rtl} models generate matrices with the same uniform distribution, but their randomly fixed non-zero elements can still produce some cache \glspl{hit}.
Furthermore, matrices generated with uniform distributions exhibit low $d_\mathit{cd}$ values (between 13\% and 14\%, see Appendix~\ref{app:mat_list}), placing them in a regime where \glspl{eviction} occur in bursts.
Under these burst-heavy conditions, the \acrshort{rtl} implementation cannot effectively leverage its out-of-order execution capabilities (see next section).

\begin{highlight}{Conclusion}{Red}
    Across synthetic \gls{banded} matrices, all three simulation models (analytical, Python, \acrshort{rtl}) demonstrate consistent memory traffic results, with only minor differences as matrices approach the \gls{PBanded} regime, without affecting the conclusions.
\end{highlight}

However, output memory traffic shows notable differences: \acrshort{rtl} achieves better traffic reduction than Python, 1.3\tm for \acrshort{redc} (vs. 1.0\tm in Python) and 5.0\tm for \acrlong{spdc} (vs. 4.1\tm in Python).
Comparing raw traffic values between \acrshort{rtl} and Python simulations across all configurations, we observe approximately 25\% lower total memory traffic in \acrshort{rtl}.
Output memory traffic varies by $\pm$15\%, with \acrshort{rtl} showing lower results for both \acrshort{redc} and \acrlong{spdc}.
Indeed, the \acrshort{rtl} model manages \glspl{reduction} out-of-order using the \textit{replay table}, whereas the Python model simulates them sequentially without overlap.
For instance, in the Python model, a \acrshort{red} triggering an \gls{eviction} completes entirely before processing the next \acrshort{red}.
Conversely, in the \acrshort{rtl} model, a \acrshort{red} triggering an \gls{eviction} can be queued in the \textit{replay table} if the \textit{miss handler} or \textit{accumulation table} reach capacity, allowing the cache to accept subsequent \acrshort{red} operations.
If a later \acrshort{red} \glspl{hit} the cache-line that would have been evicted by an earlier operation, an unnecessary \gls{eviction} is avoided, an event that would have been counted in the Python model.
This out-of-order execution capability, inherited from HPDcache, thus contributes to reduced memory traffic in the \acrshort{rtl} implementation.
