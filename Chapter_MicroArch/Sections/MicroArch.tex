\section{SpDCache Microarchitecture}
\label{sec:arch:microarch}

\lettrine{S}{pDCache} is implemented into an L1 data cache for a CVA6 RISC-V core, building upon an existing open-source high-performance set-associative data cache known as HPDcache~\cite{Fuguet2023_HPDcache}.
In contrast to traditional data caches, HPDcache introduces new features, including buffering techniques for memory writes, set-associative storage to manage read \gls{miss} requests, and dynamic configurability of each cache-line to operate in either write-back or write-through modes.
It is designed for high performance, utilizing a three-stage \gls{pipeline} to handle multiple instructions while minimizing stall potential through out-of-order cache execution.

An overview of \acrlong{spdc}'s microarchitecture is shown in Figure~\ref{fig:arch:microarch}, where \textit{black} and \textit{blue} components are originally from HPDcache.
The first stage (\textit{st0}) decodes instructions and issues reads to the cache directory and data \acrshortpl{ram} (\textit{Dir} and \textit{Data}).
These reads require one cycle to respond.
In the second stage (\textit{st1}), the response indicates whether a \gls{hit} or \gls{miss} occurred and if an \gls{eviction} is necessary.
The instruction proceeds to the final stage (\textit{st2}) only if a \gls{miss} or \gls{eviction} must be processed.
At \textit{st0}, the instruction is not yet consumed.
Depending on the response and the availability of cache resources at \textit{st1}, a \acrfull{nop} may be inserted in the \gls{pipeline}, and the instruction may be placed in a \textit{replay table}.
The instruction is then replayed with the highest priority once its dependencies are cleared.
If the instruction does not need to be replayed at \textit{st1}, it is consumed and processed.
This \gls{pipeline} serves as the foundation for implementing \acrlong{spdc}'s \gls{reduction} operations.

    \caption{\acrlong{spdc} Microarchitecture, based on HPDcache}
    \label{fig:arch:microarch}
\end{figure}

The \textit{cache controller} coordinates several auxiliary units.
The \textit{miss handler}, allocated from \textit{st2}, tracks outstanding memory read requests and notifies the cache controller with a \textit{refill} request when responses arrive.
An optional \textit{write buffer} gathers recent write to memory, allowing merges of transactions that target the same cache-line.
The \textit{flush controller}, also allocated from \textit{st2}, reads cache-line data from the \textit{Data} \acrshortpl{ram} and issues memory writes.
It temporarily stall the \gls{pipeline} to prevent \acrshort{ram} conflicts.
HPDcache further provides an \textit{UC} unit to handle uncached transactions.
Finally, the \textit{\acrshort{cmo} handler} performs internal cache operations (e.g., invalidations and flushes).
When active it takes control of the cache and can invoke the \textit{flush controller} if necessary.
Most of these units are reused and extended to implement \acrlong{spdc}.

HPDcache provides extensive static and dynamic configurability, making it a flexible foundation for further extensions.
To implement \acrlong{spdc} as well as a \gls{redC} baseline, we modified HPDcache to be highly configurable so that a single \acrshort{rtl} implementation can produce multiple designs.
The \acrshort{rtl} can be instantiated as a generic cache (the first baseline, \acrshort{gc}), as a \gls{redC} (the second baseline, \acrshort{redc}), or as \acrshort{spdc}, the solution presented in the previous chapters.

\begin{highlight}{Conclusion}{Red}
    \textbf{HPDcache} is an \textbf{open-source}, high-performance cache which we use as the \textbf{\acrshort{rtl} basis} for our work.
    We extended it to support \gls{reduction} operations and the \acrlong{spdc} mode while preserving its \textbf{configurability}.
\end{highlight}

\subsection{RTL Implementation of SpDCache}
\label{sec:arch:spdcache}

We configure the original HPDcache as a 4-way set-associative cache with 128 \glspl{set} and 64-byte cache-lines, resulting in a total cache size of 32~KiB.
We extend the three-stage \gls{pipeline} to handle the new \acrshort{red} instruction.
Figure~\ref{fig:arch:microarch} provides an overview of \acrlong{spdc}'s microarchitecture, highlighting new blocks in \textit{red}, modified blocks in \textit{blue}, and untouched components in \textit{black}.
The overall architecture aligns with the design presented in Chapter~\ref{cha:spdc}, adapted here for hardware implementation.
Let us briefly describe the operation of the \acrshort{drc} under three scenarios: \glspl{hit}, \glspl{miss} with available cache-lines, and \glspl{miss} requiring \gls{eviction}.

In the case of a \gls{hit}, the value of the \acrshort{red} instruction is accumulated with its corresponding value in the cache.
At \textit{st1}, the cache value, read in \textit{st0} from \textit{Data}, is accumulated with the \acrshort{red} value using a dedicated accumulator (\textit{Acc}).
The result is written back to \textit{Data} at \textit{st2}.

In the case of a \gls{miss}, if there is an available cache-line, the value of the \acrshort{red} instruction is simply stored in this free entry of the cache.
A free cache-line is selected at \textit{st1}, and at \textit{st2}, the \acrshort{red} value is written to \textit{Data} along with zeros to reset the selected cache-line.
A write to \textit{Dir} is also performed to change the state of the cache-line from free to valid.

If a \gls{miss} occurs but no cache-line is free, an \gls{eviction} must be processed before handling the \acrshort{red} instruction.
The \acrshort{red} instruction is placed in the \textit{PrioRED table} to be replayed with higher priority than the \textit{replay table}.
Meanwhile, at \textit{st2}, the \textit{miss handler}, \textit{flush controller}, and \textit{accumulation table} are allocated to process the \gls{eviction}.
The \gls{eviction} is a ``fire-and-forget'' operation, meaning the \acrshort{red} instruction only needs to wait for the entry in \textit{Data} to be freed, not for the entire \gls{eviction} to finish.
The \gls{eviction} process begins by fetching and resetting the chosen 64~B cache-line from \textit{Data}, which is done in one cycle by the \textit{flush controller}.
Simultaneously, the \textit{miss handler} issues a read to the main memory.
The \textit{accumulation table} waits and temporarily stores both the old cache-line from the \textit{flush controller} and the 64~B response data from the \textit{miss handler}.
It then accumulates the cache-line with the data, word by word, and sends the 64~B results back to the main memory using the existing write interface.

With the changes described above, we have transformed \mbox{HPDcache} into a \gls{redC}.
To achieve multiple cache regions, we virtually separate the cache \acrshortpl{ram}.
To maintain HPDcache's set-associative operation, we divide the cache memory along its \glspl{set}.
Thus, each region of the cache has the same structure as the overall cache but is smaller in size: each \gls{set} contains a fixed number (the number of ways) of 64-byte cache-lines, as depicted in Figure~\ref{fig:arch:set_redirection}.
Each region can map any possible address.
To ensure that every address matches a unique \gls{set} within each region, we allocate a power-of-two number of consecutive \glspl{set} to a region.
Considering the total number of \glspl{set} $S_T$ and our region $R$ mapping on $S_R < S_T$ of them, starting at the \gls{set} $s_\mathit{offset}$, if an address matches one of the \glspl{set} of this region, no action is required.
However, if the address matches a \gls{set} $s_\mathit{addr}$ outside the region, we need to map it to a unique \gls{set} in the region such that $s_\mathit{region} = (s_\mathit{addr} \% S_R) + s_\mathit{offset}$, as illustrated in Figure~\ref{fig:arch:set_redirection}.
The three regions \acrshort{grc}, \acrshort{drc}, and \acrshort{srt} are each defined by their number of \glspl{set} $S_R$ and their first \gls{set}, $s_\mathit{offset}$.
\acrlong{spdc} can thus be reconfigured to operate as a generic cache, a traditional \gls{redC} or a region-based \gls{redC} based on the needs of the application (\textit{config} in Figure~\ref{fig:arch:microarch}).

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{Chapter_MicroArch/Figures/PDF/SpDC_SetsRedirection.pdf}
    \caption{Set Redirection Depending on the Region}
    \label{fig:arch:set_redirection}
\end{figure}

The \acrshort{srt} region requires additional adjustments to function as an element-wise fully-associative cache.
This region operates as if all the \glspl{set} were combined into a single \gls{set}, containing numerous elements.
Each element requires its own directory entry.
Therefore, we replace the \textit{Dir} \acrshortpl{ram} of this region with a \acrshort{cam} (\textit{Content-Addressable Memory}) containing the addresses of each valid element in the \acrshort{srt} region.
With the \acrshort{cam}, we can check for a \gls{hit} across the entire region and retrieve the matching value stored in the \textit{Data} \acrshortpl{ram}.
We also added a queue (\textit{RMWQ}) to manage the element-wise \gls{rmw} memory transactions.

To avoid address duplication between regions, the \textit{MultiHit fetcher} in Figure~\ref{fig:arch:microarch} manages coherency between the \acrshort{drc} and \acrshort{srt}, following the description in Chapter~\ref{cha:spdc}.
Since software selects the region, this can be necessary.
The \textit{MultiHit fetcher} performs two tasks.
First, given an address, it probes the \acrshort{cam} for entries whose cache-line address matches.
If one or more matches are found, it records their positions locally and notifies the \textit{cache controller}.
Second, when activated, the \textit{MultiHit fetcher} gathers the corresponding elements from the \textit{Data} \acrshortpl{ram} of the \acrshort{srt} and assembles them into a single 64~B line that will be written into the \acrshort{drc}.

Contrary to the description in Chapter~\ref{cha:spdc}, we do not transfer isolated elements from the \acrshort{drc} to the \acrshort{rmwq} on \gls{eviction}.
In our Python simulations we observed that only a negligible fraction of elements were transferred (mean 0.3\%, range 0 to 5\% across the 48 matrices), which demonstrates the effectiveness of the \acrshort{drc} blocking behavior.
To simplify the \acrshort{rtl} implementation, this transfer is therefore omitted.

\begin{highlight}{Conclusion}{Red}
    The \acrlong{spdc} implementation introduces two primary extensions.
    First, it integrates \textbf{support for \gls{reduction} instructions (\acrshortpl{red})} into the cache \gls{pipeline} while preserving the generic cache behavior.
    This entails mechanisms for accumulating entire cache-lines and for handling element-wise atomic \acrshort{rmw} transactions.
    Second, it implements a \textbf{virtual partitioning of the cache \acrshortpl{ram}}: the memory is logically split and a \gls{set}-redirection scheme is applied to dynamically create multiple regions.
\end{highlight}

\subsection{Software Interface}
\label{sec:arch:sw_interface}

We have added several custom instructions to the CVA6 RISC-V core~\cite{Waterman2016_RISCV_ISA}.
The primary addition is the \gls{reduction} instruction (\acrshort{red}), which takes an address and a value as arguments and expects no response.
This instruction is accompanied by a region instruction (\acrshort{reg}) that specifies in which region subsequent \glspl{reduction} should be processed.
We have also added an instruction to flush the \gls{reduction} regions (\acrshort{drc} and \acrshort{srt}) of the cache, which is necessary at the end of the algorithm.
It is performed in the cache by the \textit{\acrshort{cmo} handler}.
Finally, we have included an instruction to enable and disable the \gls{reduction} mode of \acrlong{spdc}.

We demonstrate a simple example in Listing~\hyperlink{hl0}{1} that accumulates multiple values at a single address.
Before executing \acrshort{red} transactions, the \gls{reduction} engine must be enabled (line~3).
The cache transitions from a fully generic state to \acrshort{spdc} with multiple regions using a hardware-defined static configuration.
This example assumes a configuration with three regions: \acrshort{grc}, \acrshort{drc}, and \acrshort{srt}.
For a simpler \gls{redC} configuration with only two regions (\acrshort{grc} and \acrshort{drc}), the \acrshort{reg} instruction would be unnecessary.
Additionally, the instruction can accept a dynamic configuration argument specifying word granularity and operation type.
A future optimization would allow passing the entire configuration as a parameter, enabling dynamic configuration of \acrshort{spdc} regions at runtime.
Next, at line~8, we select the \acrshort{ibr} region using the \acrshort{reg} instruction.
The subsequent two \acrshort{red} instructions are then executed and stored in the \acrshort{drc}.
At line~13, we switch to the \acrshort{obr} region, causing the following \acrshort{red} instruction to be processed by the \acrshort{srt}.
Once all \acrshort{red} operations complete, \acrlong{spdc} is flushed at line~17 before disabling the reduction engine, reverting to standard cache operation.

We now describe the scenarios when the region argument is \acrshort{ibr}.
In Figure~\ref{fig:arch:pipeline_01}, the \gls{pipeline} issues reads to the \textit{Dir} and \textit{Data} \acrshortpl{ram} at \textit{st0}.
Responses arrive one cycle later at \textit{st1}, indicating whether a \gls{hit} or \gls{miss} occurs in the \acrshort{drc}.
This initial \gls{pipeline} stage is identical across all scenarios.
In case \ding{192}, a \gls{hit} is detected, so the data received at \textit{st1} is accumulated with the \acrshort{red} value and forwarded to \textit{st2} for write-back to the \textit{Data} \acrshort{ram}.

When a \gls{miss} occurs, multiple scenarios are possible (Figures~\ref{fig:arch:pipeline_02} to~\ref{fig:arch:pipeline_05}), as an \gls{eviction} may be required and the \textit{MultiHit Fetcher} may be active.
In case \ding{193}, upon detecting a \gls{miss} at \textit{st1}, the \gls{pipeline} checks the \textit{Dir} \acrshort{cam} of the \acrshort{srt} for a ``multi-\gls{hit}''.
As described in Chapter~\ref{cha:spdc}, when a \gls{miss} occurs in the \acrshort{drc}, we gather elements with the same cache-line address from the \acrshort{srt} to prevent duplication.
This check completes within the cycle and notifies the \textit{MultiHit Fetcher}.
If a ``multi-\gls{hit}'' is found, the \textit{MultiHit Fetcher} allocates an entry and takes control of the \gls{pipeline} to read and invalidate matching entries from the \textit{Data} \acrshort{ram}, storing them locally for later use.
If no ``multi-\gls{hit}'' occurs (case \ding{193}), we handle a simple \gls{miss} by writing the \acrshort{red} value to a free cache-line in the \textit{Data} \acrshort{ram} at \textit{st2}.

In case \ding{195}, a ``multi-\gls{hit}'' is detected and the \textit{MultiHit Fetcher} is allocated.
The \acrshort{red} instruction is placed in the \textit{PrioRED Table} for immediate replay once the \gls{pipeline} becomes available, proceeding to case \ding{194} or \ding{196} depending on whether an \gls{eviction} is needed.

In case \ding{196}, which occurs only upon replay, the \acrshort{red} instruction \glspl{miss} in the \acrshort{drc} but \glspl{hit} in the \textit{MultiHit Fetcher}, which has cached the gathered data from the previous case \ding{195}.
The \acrshort{red} value is then accumulated with the \textit{MultiHit Fetcher} data and written to the \textit{Data} \acrshort{ram} at \textit{st2}.

Case \ding{194} handles the \gls{eviction} scenario, which applies whether or not a ``multi-\gls{hit}'' replay is involved.
When an \gls{eviction} is required, a victim cache-line is selected at \textit{st1} and must be evicted before inserting the new value.
The \gls{eviction} process begins at \textit{st2}, with the \textit{Miss Handler}, \textit{Flush Controller}, and \textit{Accumulation Table} allocated for the victim address.
Concurrently, the \acrshort{red} instruction is placed in the \textit{PrioRED Table} for immediate replay.
Once the victim cache-line is freed, the instruction is replayed as case \ding{193} or \ding{196}, depending on whether data exists for the same cache-line address in the \textit{MultiHit Fetcher}.
Note that when a ``multi-\gls{hit}'' occurs (case \ding{195}), the \textit{MultiHit Fetcher} retains the gathered data until reaching case \ding{196}.
Subsequently, since the data is no longer present in the \textit{Dir} \acrshort{cam} and \textit{Data} \acrshort{ram}, no ``multi-\gls{hit}'' will occur for that address on replay.

Figure~\ref{fig:arch:pipeline_06} depicts the \gls{eviction} process, which executes in parallel with the \gls{pipeline}.
Upon receiving the victim address in case \ding{194}, the \textit{Flush Controller} immediately reads the \textit{Data} \acrshort{ram} and invalidates the entry in a single cycle.
This allows the \gls{pipeline} to replay the \acrshort{red} instruction without waiting for the entire \gls{eviction} to complete.
The \textit{Flush Controller} then forwards the victim data to the \textit{Accumulation Table} for temporary storage.
Meanwhile, the \textit{Miss Handler} issues a memory read, which may take up to one hundred cycles in worst-case scenarios.
Upon receiving the data from memory, the \textit{Miss Handler} sends it to the \textit{Accumulation Table}.
Once the \textit{Accumulation Table} has both data lines, it performs word-by-word accumulation and returns the result to the \textit{Flush Controller}, which writes it back to memory to complete the \gls{eviction}.

\noindent The following sequences enumerate all possible cases when the region argument is \acrshort{ibr}:
\begin{itemize}
    \item \gls{hit} in the \acrshort{drc}: \ding{192};
    \item \gls{miss} in the \acrshort{drc}: \ding{193};
    \item \gls{miss}+\gls{eviction} in the \acrshort{drc}: \ding{194} + \ding{197} $\rightarrow$ \ding{193};
    \item \gls{miss} in the \acrshort{drc} with ``multi-\gls{hit}'' in the \acrshort{srt}: \ding{195} $\rightarrow$ \ding{196};
    \item \gls{miss}+\gls{eviction} in the \acrshort{drc} with ``multi-\gls{hit}'' in the \acrshort{srt}: \ding{195} $\rightarrow$ \ding{194} + \ding{197} $\rightarrow$ \ding{196}.
\end{itemize}

We now describe the cases when the region argument is \acrshort{obr}, as illustrated in Figures~\ref{fig:arch:pipeline_07} to~\ref{fig:arch:pipeline_10}.
In all cases, we check for \glspl{hit} in both the \acrshort{drc} and \acrshort{srt} at \textit{st0}.
If a \gls{hit} occurs in the \acrshort{drc}, case \ding{198}, the \acrshort{red} instruction is processed in the \acrshort{drc} regardless of the region argument.
It is then sent to the \textit{PrioRED Table} for replay as a \acrshort{drc} \gls{hit}, corresponding to case \ding{192}.

If no \gls{hit} is found in the \acrshort{drc}, we check the \textit{Dir} \acrshort{cam} for \glspl{hit} in the \acrshort{srt}.
In case of a \gls{hit}, \ding{199}, the data is accumulated with the \acrshort{red} value and written back to the \textit{Data} \acrshort{ram}.
In case of a \gls{miss}, \ding{200}, the data is inserted into the \textit{Data} \acrshort{ram}.
For an \gls{eviction}, \ding{201}, the victim data is read directly between \textit{st0} and \textit{st1}, since the \textit{Dir} \acrshort{cam} responds within one cycle and can initiate the victim read before \textit{st1}.
Thus, the \acrshort{red} value is inserted into the \textit{Data} \acrshort{ram} at \textit{st2}, while the victim data is sent to the \acrshort{rmwq}.
The \acrshort{rmwq} independently issues atomic \acrshort{rmw} transactions to memory as entries become available.

\noindent The following sequences enumerate all possible cases when the region argument is \acrshort{obr}:
\begin{itemize}
    \item \gls{hit} in the \acrshort{drc}, \ding{198} $\rightarrow$ \ding{192};
    \item \gls{hit} in the \acrshort{srt}, \ding{199};
    \item \gls{miss} in the \acrshort{srt}, \ding{200};
    \item \gls{miss}+\gls{eviction} in the \acrshort{srt}, \ding{201}.
\end{itemize}

\begin{highlight}{Note}{SkyBlue}
    This implementation leverages most of the original \gls{pipeline} capabilities while keeping modifications to a minimum.
    We do not claim this is the optimal design for \acrlong{spdc}.
    A ground-up redesign or significant modifications to HPDcache could yield better performance.
\end{highlight}

\begin{highlight}{Conclusion}{Red}
    We implemented \textbf{\acrlong{spdc}} in \acrshort{rtl} by extending HPDcache with \textbf{minimal modifications}, introducing support for \gls{reduction} operations and multiple cache regions without requiring additional memory.
    This \textbf{configurable design} allows a single \acrshort{rtl} implementation to function as a generic cache, a traditional \gls{redC}, or the region-based \acrlong{spdc} variant, facilitating comprehensive performance evaluation.
\end{highlight}
